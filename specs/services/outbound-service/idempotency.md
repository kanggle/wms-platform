# outbound-service â€” Idempotency Strategy

Idempotency guarantees for `outbound-service`, covering four independent
mutation surfaces: synchronous REST, ERP order webhook ingestion, Kafka event
consumption (saga steps + master snapshots), and saga-level state-machine
idempotency. This document is the single reference for idempotency
implementation in this service.

Relevant rules: trait `transactional` T1 (REST idempotency), T8 (eventId
dedupe), T2 (no distributed TX), T4 (no direct status update), T6
(compensation), T7 (saga); trait `integration-heavy` I4 (idempotent
side-effects), I6 (webhook reception pattern).

> **Implementation status (TASK-BE-051, 2026-05-11)**: REST idempotency
> (Â§1) is wired end-to-end via {@code OutboundIdempotencyFilter} +
> {@code IdempotencyStore} port + {@code RedisIdempotencyStore} adapter
> + {@code InMemoryIdempotencyStore} ({@code standalone} profile fallback).
> The filter runs at {@code HIGHEST_PRECEDENCE + 20}, after Spring Security
> and before DispatcherServlet, applying to {@code POST/PATCH/PUT/DELETE
> /api/v1/outbound/**} except {@code /webhooks/**}. The previous behaviour
> ({@code RequestContext.requireIdempotencyKey} as a presence-only
> controller guard) is preserved as the {@code 400 VALIDATION_ERROR} path
> when the filter sees no header.

## Overview

`outbound-service` has **four** mutation surfaces that all require
idempotency:

| Surface | Mechanism | Storage | TTL |
|---|---|---|---|
| REST (POST / PATCH / DELETE) | `Idempotency-Key` header | Redis | 24h |
| ERP order webhook ingest | `X-Erp-Event-Id` header | PostgreSQL `erp_order_webhook_dedupe` | 7 days |
| Kafka consumer (saga + `master.*`) | `eventId` field in event envelope | PostgreSQL `outbound_event_dedupe` | 30 days |
| Saga state machine | current-state guard inside `OutboundSaga` aggregate | (stateful â€” derives from `outbound_saga.state`) | (lifetime of saga; terminal sagas retained per archival policy) |

These four mechanisms are independent. A REST call, a webhook, a Kafka
consumer path, and a saga transition are never deduplicated against each
other â€” they operate on different envelopes with different replay
characteristics. The saga-level layer sits on **top of** the eventId dedupe
and provides defense-in-depth: even if dedupe is bypassed (e.g., 30-day TTL
expired), the state machine still rejects already-applied transitions.

## 1. REST Idempotency (`Idempotency-Key` header)

### 1.1 Scope

All mutating endpoints in
[`specs/contracts/http/outbound-service-api.md`](../../contracts/http/outbound-service-api.md)
require `Idempotency-Key`:

| Method | Endpoints |
|---|---|
| POST | `/orders` |
| POST | `/orders/{id}:cancel` |
| POST | `/picking-requests/{id}/confirmations` |
| POST | `/orders/{id}/packing-units` |
| PATCH | `/packing-units/{id}` (seal) |
| POST | `/orders/{id}/shipments` |
| POST | `/shipments/{id}:retry-tms-notify` |

GET endpoints are naturally idempotent â€” no header required or accepted.
ERP webhook endpoint (`POST /webhooks/erp/order`) does NOT use
`Idempotency-Key` â€” see Â§2.

### 1.2 Key Format

> **Cross-service note**: `outbound-service` uses the *raw-appended*
> idempotency-key family but with a **255-char cap** (vs the 128-char
> `inventory`/`inbound`/`admin` and the 64-char hashed-flat `master`). This
> wider cap is an intentional saga-flow accommodation; the full WMS
> divergence is documented authoritatively in
> [`master-service/idempotency.md` Â§ Cross-Service Idempotency Key Conventions](../master-service/idempotency.md#cross-service-idempotency-key-conventions-intentional-divergence)
> (TASK-BE-293 WI-3, decision B).

- Type: UUID v4 (recommended) or any opaque string â‰¤ **255 chars** (per
  trait `transactional` T1 ceiling). Length over â†’ `400 VALIDATION_ERROR`
  with the message `Idempotency-Key header exceeds 255 characters`. The
  filter validates the header length before the body-hash step.
- Generated by the **caller** (gateway client UI, integration test, ops tool)
- Must be unique per logical operation. Reuse across different logical
  operations triggers `DUPLICATE_REQUEST` (409)
- **Namespace**: the storage key (and therefore the dedupe scope) is
  `(method, path, idempotencyKey)`. Reusing the same `Idempotency-Key`
  across different endpoints (or different HTTP methods on the same path)
  is independent â€” the filter computes
  `{METHOD}:{sha256hex(requestURI)}:{idempotencyKey}` so two requests to
  different endpoints never share an entry.

### 1.3 Redis Storage Layout

Redis key pattern: `outbound:idempotency:{method}:{path_hash}:{idempotency_key}`

Where:
- `{method}`: HTTP method (`POST`, `PATCH`, `PUT`, `DELETE`) â€” uppercase
- `{path_hash}`: SHA-256 hex digest of `request.getRequestURI()` â€” UTF-8
  encoded. Includes path variables â€” so `/orders/abc:cancel` and
  `/orders/xyz:cancel` get distinct keys.
- `{idempotency_key}`: the raw `Idempotency-Key` header value (after
  length validation)

The lock-key sibling lives under `outbound:idempotency:lock:{...same suffix...}`
so entry + lock collocate in Redis log output but never collide.

Value stored (JSON, serialised by Jackson):
- `requestHash` â€” SHA-256 hex of the canonical request body (key-sorted +
  whitespace-normalised; see Â§1.6)
- `status` â€” HTTP status code (only 2xx is stored)
- `bodyJson` â€” raw response body as UTF-8 text
- `contentType` â€” for the replay path
- `createdAt` â€” instant the entry was written

TTL: 86400 seconds (24h) from first write, applied via `SET key value EX 86400`.
Lock TTL: 30 seconds, applied via `SET key 1 NX EX 30` for atomic acquire.

> **Spec-aligned naming.** The Redis prefix MUST be `outbound:idempotency:`
> â€” matches the convention enforced for inventory-service (TASK-BE-025) and
> inbound-service. This is a service-level constant defined in
> `RedisIdempotencyStore.ENTRY_PREFIX`.

### 1.4 Processing Logic

On receiving a mutating request:

```
1. Read Redis entry for (method, path_hash, Idempotency-Key)

2a. Entry does NOT exist:
    â†’ Store a PENDING marker (prevents concurrent duplicate)
    â†’ Execute the use-case (@Transactional)
    â†’ On success: store {statusCode, body, bodyHash} in Redis with 24h TTL
    â†’ Return fresh response

2b. Entry exists, status = PENDING (concurrent duplicate in-flight):
    â†’ Return 503 with Retry-After: 1

2c. Entry exists, status = COMPLETE:
    â†’ Compare bodyHash of new request vs stored bodyHash
    â†’ If match: return cached response (same status code + body)
    â†’ If mismatch: return 409 DUPLICATE_REQUEST

2d. Entry exists, status = FAILED:
    â†’ Treat as if entry does NOT exist â€” allow retry
    (Use-case failed previously; the caller retrying with same key is expected)
```

### 1.5 Error Responses

| Scenario | HTTP | Error Code | Notes |
|---|---|---|---|
| Header absent on POST/PATCH/DELETE | 400 | `VALIDATION_ERROR` | Controller guard via `RequestContext.requireIdempotencyKey`; the filter passes through to the controller for non-empty / present-but-blank header rejection |
| Header > 255 chars | 400 | `VALIDATION_ERROR` | Filter-level length check before body hashing |
| Key exists, body matches | 2xx | â€” | Cached response replayed |
| Key exists, body differs | 409 | `DUPLICATE_REQUEST` | Caller bug: reused key for different op. Synonymous with `IDEMPOTENCY_KEY_REUSE_INVALID` referenced in TASK-BE-051 â€” the canonical platform-common code `DUPLICATE_REQUEST` is what's emitted on the wire. |
| Lock held (PENDING â€” concurrent in-flight) | 503 | â€” | `Retry-After: 1` header set; body uses code `PROCESSING` |
| Key in FAILED | (retry fresh) | â€” | Prior failure was not idempotent; the filter only stores 2xx responses, so a 4xx/5xx response leaves the slot empty for retry |
| Redis unavailable (lookup / lock fails) | (passes through) | â€” | Fail-open per Â§ Failure Scenarios; logged + `outbound.idempotency.store.failure` counter incremented |

### 1.6 Body Hash

Canonical body for hashing:
- Parse JSON body
- Recursively sort object keys alphabetically
- Re-serialize to compact JSON
- SHA-256 (hex string)

Purpose: detect caller sending the same key with a modified body (clock skew,
client bug, retry with different quantity). The hash comparison catches this
before a second mutation is applied.

### 1.7 Domain-Level Backstops

Multiple domain-level uniqueness constraints back up the Redis dedupe â€” they
remain effective after Redis TTL expiry:

| Endpoint | Domain backstop | Failure code |
|---|---|---|
| `POST /orders` (manual) | `order.order_no` UNIQUE constraint â€” same `orderNo` retried after Redis eviction | `409 ORDER_NO_DUPLICATE` |
| `POST /orders` (any) | `outbound_saga.order_id` UNIQUE constraint â€” at most one saga per order | `409 ORDER_ALREADY_HAS_SAGA` (manifests as `STATE_TRANSITION_INVALID`) |
| `POST /picking-requests/{id}/confirmations` | `picking_confirmation.picking_request_id` UNIQUE â€” at most one confirmation per request; second attempt sees `PickingRequest.status != SUBMITTED` | `STATE_TRANSITION_INVALID` |
| `POST /orders/{id}/shipments` | `shipment.order_id` UNIQUE â€” one shipment per order; second attempt sees `Order.status != PACKED` | `STATE_TRANSITION_INVALID` |
| `POST /orders/{id}:cancel` | `Order.status` guard â€” second cancel sees `CANCELLED` not eligible state | `STATE_TRANSITION_INVALID` (`ORDER_ALREADY_SHIPPED` after `SHIPPED`) |
| `PATCH /packing-units/{id}` (seal) | `packing_unit.status` guard â€” second seal sees `SEALED` | `STATE_TRANSITION_INVALID` |
| `POST /shipments/{id}:retry-tms-notify` | `shipment.tms_status` guard â€” endpoint is naturally idempotent: `NOTIFIED` shipments return cached ack; only `NOTIFY_FAILED` triggers a new attempt | `STATE_TRANSITION_INVALID` if `tms_status = PENDING` (saga not yet at SHIPPED) |

These two-layer defenses ensure outbound idempotency survives Redis TTL
expiry. Most retries happen within 24h; the domain layer is the safety net.

## 2. Webhook Idempotency (`X-Erp-Event-Id`)

### 2.1 Scope

The single endpoint `POST /webhooks/erp/order`. See
[`specs/contracts/webhooks/erp-order-webhook.md`](../../contracts/webhooks/erp-order-webhook.md)
Â§ Replay Dedupe for the wire-level contract.

### 2.2 ErpOrderWebhookDedupe Table

Declared in
[`domain-model.md`](domain-model.md) Â§9. Physical table:

```sql
CREATE TABLE erp_order_webhook_dedupe (
  event_id     VARCHAR(80) PRIMARY KEY,
  received_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Retention: 7 days. Nightly cron deletes rows where
`received_at < now() - interval '7 days'`. 7 days covers ERP retry windows
(usually 1â€“24h) plus operator buffer.

### 2.3 Processing Logic â€” Synchronous (Controller)

Within the same `@Transactional` boundary as the inbox write:

```
1. INSERT INTO erp_order_webhook_dedupe (event_id, received_at)
   VALUES (?, now())
   ON CONFLICT (event_id) DO NOTHING
   RETURNING event_id

2a. Row inserted (returned):
    â†’ INSERT INTO erp_order_webhook_inbox (event_id, raw_payload, signature, received_at, status)
      VALUES (?, ?, ?, now(), 'PENDING')
    â†’ Commit TX
    â†’ Respond 200 {status: "accepted", eventId, orderNo, receivedAt}

2b. No row returned (conflict â€” already exists):
    â†’ Lookup previously stored receivedAt for the response body
    â†’ No inbox write (already exists from prior receipt)
    â†’ Respond 200 {status: "ignored_duplicate", eventId, previouslyReceivedAt}
```

The dedupe table is the **source of truth** for "did we already see this
event-id". The inbox is the **work queue**; in steady state, every dedupe
row has a corresponding inbox row, but the dedupe row alone is sufficient
for duplicate detection.

### 2.4 Processing Logic â€” Asynchronous (Background Processor)

The background processor runs every 1s, scans `erp_order_webhook_inbox` for
`status = PENDING` rows, and applies the domain logic via
`ReceiveOrderUseCase`. The processor is already idempotent because:

- The inbox has the row; running `ReceiveOrderUseCase` succeeds â†’ `status =
  APPLIED`. A future re-scan finds no `PENDING` rows for this event-id.
- If the processor crashes mid-apply: the TX rolled back. The inbox row stays
  `PENDING`. Next scan picks it up.
- If `ReceiveOrderUseCase` fails (e.g., master snapshot stale, partner
  inactive): the inbox row moves to `status = FAILED` with `failure_reason`.
  The processor does NOT retry FAILED rows automatically â€” ops triggers
  retry via Â§2.5.

`ReceiveOrderUseCase` is the unifying entry-point for both webhook ingest
and the synchronous manual REST path (`POST /orders`). It atomically
creates: `Order` (status=`RECEIVED` then immediately `PICKING` via
`startPicking()`) + `OutboundSaga` (state=`REQUESTED`) + outbox rows
(`outbound.order.received` and `outbound.picking.requested`) â€” all in one
`@Transactional` boundary.

### 2.5 Manual Retry of FAILED Inbox Rows

In v1: not exposed via REST. Ops investigates via SQL or admin dashboard
(read-only), and re-emission flows back through the standard ERP retry path
(after the 7-day dedupe TTL has expired, or by direct DB row deletion if
the team agrees).

In v2: `POST /api/v1/outbound/webhooks/inbox/{eventId}:retry` will reset
`status = PENDING` and the processor picks it up on next scan.

### 2.6 Webhook vs REST Crossover

`source` field on `Order` distinguishes the two paths (`MANUAL` vs
`WEBHOOK_ERP`). They share the same `order_no` UNIQUE constraint:

- ERP webhook arrives, creates `Order(order_no="ORD-20260501-001",
  source=WEBHOOK_ERP)`.
- An operator separately creates an order with the same `orderNo` via
  `POST /orders` â†’ 409 `ORDER_NO_DUPLICATE`.

So the two paths cannot accidentally double-create. The reverse (operator
creates `MANUAL` first, ERP webhook arrives later) inverts the scenario but
results in the same: the second producer hits the unique constraint, the
inbox row marks `FAILED` with reason `ORDER_NO_DUPLICATE`.

## 3. Kafka Consumer Idempotency (EventId Dedupe)

### 3.1 Scope

All Kafka consumers in `outbound-service`. v1 consumes both saga-step events
from `inventory-service` and master snapshot events from `master-service`:

| Consumer | Topic | Purpose |
|---|---|---|
| `InventoryReservedConsumer` | `wms.inventory.reserved.v1` | Saga step 1 success â†’ `RESERVED` |
| `InventoryReleasedConsumer` | `wms.inventory.released.v1` | Compensation acknowledged â†’ `CANCELLED` |
| `InventoryConfirmedConsumer` | `wms.inventory.confirmed.v1` | Saga step 4 success â†’ `COMPLETED` |
| `InventoryAdjustedConsumer` (filtered: `reason=INSUFFICIENT_STOCK`) | `wms.inventory.adjusted.v1` | Out-of-stock signal â†’ `RESERVE_FAILED` |
| `MasterWarehouseConsumer` | `wms.master.warehouse.v1` | MasterReadModel snapshot refresh |
| `MasterZoneConsumer` | `wms.master.zone.v1` | Same |
| `MasterLocationConsumer` | `wms.master.location.v1` | Same |
| `MasterSkuConsumer` | `wms.master.sku.v1` | Same |
| `MasterLotConsumer` | `wms.master.lot.v1` | Same |
| `MasterPartnerConsumer` | `wms.master.partner.v1` | Same |

`outbound-service` does NOT consume any `inbound.*` events in v1.

### 3.2 OutboundEventDedupe Table

Declared in [`domain-model.md`](domain-model.md) Â§8. Physical table:

```sql
CREATE TABLE outbound_event_dedupe (
  event_id     UUID        PRIMARY KEY,
  event_type   VARCHAR(60) NOT NULL,
  processed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  outcome      VARCHAR(20) NOT NULL  -- 'APPLIED' | 'IGNORED_DUPLICATE' | 'FAILED'
);
```

Retention: 30 days. A nightly cron deletes rows where `processed_at < now() -
interval '30 days'`. 30 days covers operator-initiated DLQ replays and
broker retention windows.

### 3.3 Processing Logic

Within the same `@Transactional` boundary as the saga / snapshot mutation.
The `EventDedupePort` adapter declares `Propagation.MANDATORY` so the
dedupe write joins the consumer's TX (matches the pattern from
inventory-service TASK-BE-027 and inbound-service):

```
1. INSERT INTO outbound_event_dedupe (event_id, event_type, processed_at, outcome)
   VALUES (?, ?, now(), 'APPLIED')
   ON CONFLICT (event_id) DO NOTHING
   RETURNING event_id

2a. Row inserted (returned):
    â†’ For saga consumers: SagaCoordinator.apply(event)
    â†’ For master consumers: MasterReadModel UPSERT
    â†’ Commit TX (mutation + dedupe row in same TX)

2b. No row returned (conflict â€” already exists):
    â†’ Skip mutation
    â†’ Commit TX (no-op; the IGNORED_DUPLICATE outcome is already recorded
      from the prior delivery)
    â†’ Log: INFO "Duplicate event skipped" with {eventId, eventType, consumer}
```

### 3.4 Master Snapshot Consumer Layered Idempotency

Master snapshot upserts are doubly idempotent â€” the table-level UPSERT with
`master_version` guard is naturally idempotent on its own:

```sql
INSERT INTO partner_snapshot (id, ..., master_version)
VALUES (?, ..., ?)
ON CONFLICT (id) DO UPDATE
  SET status = EXCLUDED.status, ..., master_version = EXCLUDED.master_version,
      cached_at = now()
  WHERE partner_snapshot.master_version < EXCLUDED.master_version;
```

If `master_version <= cached`, the UPDATE clause matches no rows â€”
naturally idempotent.

The `outbound_event_dedupe` row is still written for two reasons:

1. **Observability**: the `outcome` field tells ops "did we just no-op a
   duplicate?" in a queryable way. The UPSERT alone doesn't surface that.
2. **Defense in depth**: future master events may carry payloads that
   require non-idempotent side effects (e.g., counter increments) where the
   dedupe table's PK is the only line of defense.

### 3.5 Saga-Step Consumer Layered Idempotency

Saga consumers (`InventoryReservedConsumer`, `InventoryReleasedConsumer`,
`InventoryConfirmedConsumer`, `InventoryAdjustedConsumer`) are protected by
**two** layers:

1. **EventId dedupe** â€” the same Kafka envelope `eventId` is processed
   exactly once (this section).
2. **Saga state-machine guard** â€” even if dedupe is bypassed (30-day TTL
   expired, operator-replayed message from raw Kafka offsets, eventId
   collision across topics), the saga aggregate refuses already-applied
   transitions silently. See Â§4.

This is the load-bearing safety property: saga progression is **monotonic**,
and re-delivery at any layer is harmless.

### 3.6 Retry and DLQ

Per trait `integration-heavy` rule I5:

- Spring Kafka `DefaultErrorHandler` with exponential backoff:
  - Retry 1: 1s delay
  - Retry 2: 2s delay
  - Retry 3: 4s delay
  - After 3 retries: route to `<topic>.DLT` via
    `DeadLetterPublishingRecoverer`
- DLT topic name pattern: `wms.inventory.reserved.v1.DLT` (Spring Kafka
  default suffix `.DLT`)
- DLT messages include original headers + `kafka_dlt-exception-*` headers
  for inspection

Permanent failures (unparseable JSON, unknown `eventType`): go to DLT
immediately (no retry). Non-retryable exception list includes
`JsonProcessingException` and custom `UnknownEventTypeException`.

Transient failures (DB connection, optimistic lock on `outbound_saga`):
retried per backoff schedule. Optimistic-lock conflicts on saga transitions
are typical when a REST cancel and a Kafka reply arrive close together â€”
the failing message goes back into the consumer queue and re-applies on
re-poll, where the saga state machine resolves the now-invalid transition
(see Â§4.3).

### 3.7 Consumer Group

All consumers in `outbound-service` share consumer group ID:
`outbound-service`.

Partition key for inventory reply events is `sagaId` (per
`architecture.md` Â§ Event Consumption). This preserves in-order delivery
within a single saga: `inventory.reserved` for sagaA always arrives before a
hypothetical second event for sagaA, even under partition reassignment.

## 4. Saga-Level Idempotency (State-Machine Guard)

### 4.1 Why a Fourth Layer

The eventId dedupe (Â§3) handles re-delivery from the **broker**. The
state-machine guard handles re-delivery from **anywhere else**:

- 30-day TTL purged the dedupe row, then an operator replayed an old DLT
  message from raw offsets.
- A saga sweeper (`architecture.md` Â§ Saga Sweeper) re-emits
  `outbound.picking.requested` for a stuck saga that actually already
  reserved â€” the inventory-service consumer responds with
  `inventory.reserved` *again*. The new event has a fresh `eventId` (so
  dedupe doesn't catch it) but the saga is already at `RESERVED`.
- A bug in compaction / dual-write surfaces an event with a duplicated
  business key but a fresh `eventId`.

In all cases, the saga aggregate **must** refuse the transition without
error. This is the load-bearing correctness property of a choreographed
saga.

### 4.2 Implementation

`OutboundSaga.apply(event)` â€” implemented as a domain method on the
aggregate â€” checks the current state against the event:

```
public void apply(SagaEvent event) {
    SagaState target = stateMachine.targetFor(this.state, event.kind());
    if (target == null) {
        // No legal transition from current state for this event kind.
        // Two cases:
        //   (a) already-applied (re-delivery): silent no-op
        //   (b) genuinely invalid (e.g., inventory.confirmed before SHIPPED): log + alert
        if (stateMachine.isAlreadyApplied(this.state, event.kind())) {
            log.info("saga_event_already_applied", ...);
            return;  // no-op, no exception
        } else {
            log.warn("saga_event_invalid_transition", ...);
            throw new StateTransitionInvalidException(this.state, event.kind());
        }
    }
    this.state = target;
    this.lastTransitionAt = Instant.now();
    this.version++;  // optimistic lock bump
}
```

`isAlreadyApplied(currentState, eventKind)` returns `true` for these pairs:

| Current state | Event kind | Reason |
|---|---|---|
| `RESERVED`, `PICKING_CONFIRMED`, `PACKING_CONFIRMED`, `SHIPPED`, `COMPLETED` | `inventory.reserved` | Already past reserve step |
| `COMPLETED` | `inventory.confirmed` | Already past confirm step |
| `CANCELLED` | `inventory.released` | Already past release step |
| `RESERVE_FAILED`, `CANCELLED` | `inventory.confirmed` | Saga is terminal in failure state â€” confirmed cannot apply (this is logged WARN as anomaly but treated as no-op for safety) |

Any other "no legal transition" combination (e.g., `inventory.confirmed`
arriving when saga is in `REQUESTED`) is a genuine impossible state â€” log
WARN + raise a metric, but **silently drop** to avoid poison-pilling the
consumer (the message will go to DLT after retries; ops investigates).

### 4.3 Race: REST Cancel vs Inventory Reply

A frequent race during testing and production:

```
T0: saga = RESERVED, version V
T1: REST POST /orders/{id}:cancel arrives â†’ loads saga at V â†’ emits cancel
T2: inventory.released event consumed â†’ loads saga at V â†’ emits transition
T3: Whoever commits first wins at V+1
T4: Loser hits OptimisticLockingFailureException â†’ 409 (REST) /
    consumer-retry (Kafka)
T5: On retry, second actor loads saga at V+1 â€” state has changed
T6: State machine evaluates the transition against V+1's state:
      - For REST cancel from CANCELLATION_REQUESTED: rejected as already-applied
      - For Kafka inventory.released against CANCELLATION_REQUESTED:
        legal â†’ moves to CANCELLED
```

In this race, exactly one of the two actors observes "I caused the
transition", and the other observes "already applied". No double-mutation,
no lost event.

### 4.4 Saga Sweeper Re-emission Safety

The sweeper (`architecture.md` Â§ Saga Sweeper) re-emits driver events for
sagas stuck in transitional states. When the inventory-service receives a
re-emitted `outbound.picking.requested`:

1. Inventory's own dedupe (`inventory_event_dedupe`) sees the same
   `eventId` if within 30 days â†’ no-op.
2. If beyond 30 days, inventory's saga-side guard (the equivalent
   reservation table's UNIQUE constraint on `reservation_id` =
   `picking_request.id`) prevents double-reserve.
3. Either way, inventory emits a fresh `inventory.reserved` (or returns
   the stored ack).

When `outbound-service` consumes the (possibly-fresh) `inventory.reserved`:

1. EventId dedupe â€” fresh eventId might pass.
2. Saga state-machine guard â€” saga is already `RESERVED` â†’
   `isAlreadyApplied()` returns true â†’ silent no-op.

The fourth layer makes sweeper re-emission safe even when the eventId
dedupe doesn't have the row.

## 5. Decision Table â€” Which Mechanism Applies

| Scenario | Mechanism | Behavior |
|---|---|---|
| REST POST with same `Idempotency-Key`, same body | Redis | Replay cached 2xx |
| REST POST with same `Idempotency-Key`, different body | Redis | 409 `DUPLICATE_REQUEST` |
| REST POST `/orders`, same `orderNo`, different key, key expired | Domain unique constraint | 409 `ORDER_NO_DUPLICATE` |
| REST POST `:cancel` on already-cancelled order | Domain status guard | 422 `STATE_TRANSITION_INVALID` |
| REST POST `/orders/{id}/shipments` on already-shipped order | Domain status guard | 422 `ORDER_ALREADY_SHIPPED` |
| ERP webhook with same `X-Erp-Event-Id` | PostgreSQL `erp_order_webhook_dedupe` | 200 `ignored_duplicate`, single domain effect |
| ERP retries the same order with NEW `X-Erp-Event-Id` (operator deleted dedupe row by mistake) | Domain unique on `order_no` | inbox row â†’ status=FAILED with reason `ORDER_NO_DUPLICATE` |
| Kafka broker redelivers same `inventory.reserved` `eventId` | PostgreSQL `outbound_event_dedupe` | Skip, log as duplicate |
| Kafka broker redelivers `inventory.reserved` with **fresh** eventId for already-RESERVED saga | Saga state-machine guard | Silent no-op (4th layer) |
| Kafka DLQ replay (operator-initiated) within 30 days | PostgreSQL `outbound_event_dedupe` | Skip if already processed |
| Kafka DLQ replay after 30-day TTL purge | Saga state-machine guard | Silent no-op if already-applied; legal transition if not |
| Saga sweeper re-emits `outbound.picking.requested` after inventory already reserved | Saga state-machine guard (on receipt of re-emitted `inventory.reserved`) | No-op |
| Two pods consume same partition briefly (rebalance) | At-most-once in partition | Not possible â€” Kafka partition assigned to one consumer |
| REST `:retry-tms-notify` on already-NOTIFIED shipment | Domain status guard | 200 with cached ack (no second TMS call); naturally idempotent |

### 5.1 30-day Replay Risk for Kafka

After 30 days, the dedupe table has purged the row. An operator-initiated
replay of an `inventory.reserved` message older than 30 days would re-enter
the consumer. The saga state-machine guard (Â§4) absorbs this â€” saga is
either already past `RESERVED` (no-op) or, in the rare case it's still
`REQUESTED` because the original was lost, the replay is the recovery.

### 5.2 7-day Replay Risk for Webhooks

After 7 days, the webhook dedupe row is purged. ERP re-emission of the
same `event_id` after that window WOULD pass the dedupe check. The result
depends on the `order_no`:

- If the resulting Order is identical (same `order_no`, same content) â†’
  blocked by `order.order_no` UNIQUE constraint â†’ inbox row marks `FAILED`.
  No double effect.
- If ERP re-emits with a NEW `orderNo` (e.g., they think it's a new order)
  â†’ treated as a new Order, which is correct.

The 7-day TTL is intentional: long enough for ERP retry storms to settle,
short enough to keep the dedupe table small.

## 6. Idempotency-Key Lifecycle Guidance for Callers

### REST callers (operator UI, integration tests)

1. Generate one `Idempotency-Key` (UUID v4) at the start of the user
   action (e.g., when the user opens the "Confirm Shipping" dialog).
2. Include it on the request.
3. On retry (network failure, timeout): reuse the **same key**.
4. On receiving 2xx (fresh or cached): proceed.
5. On receiving 409 `DUPLICATE_REQUEST`: stop â€” investigate the body
   mismatch (likely a UI bug).
6. On receiving 503 (PENDING): retry after the `Retry-After` interval.

### ERP webhook callers

1. ERP generates one `X-Erp-Event-Id` per logical order event.
2. On retry (5xx, timeout, no response within ERP's expected window):
   reuse the **same** `X-Erp-Event-Id`.
3. On 200 `accepted`: ERP marks the event as delivered.
4. On 200 `ignored_duplicate`: ERP marks delivered (we already had it).
5. On 401 (signature/timestamp): ERP investigates clock drift, secret
   rotation, integration version.
6. On 422: ERP fixes the payload schema and re-emits with a NEW event-id
   (because the schema fix changes the payload â€” keeping the same id
   would be a body mismatch).
7. On 503: ERP retries with same id after backoff.

### Operator UI for `outbound-service`

1. The ops UI generates the `Idempotency-Key` before showing the
   confirmation dialog.
2. If the user double-clicks: second submission reuses the key â†’ cached
   response returned.
3. If the user leaves and returns: a new key is generated â†’ a new
   independent operation (correct: user intentionally re-doing the
   action).

### Saga sweeper / internal background jobs

1. Sweeper re-emission paths do NOT use REST â€” they emit Kafka events
   directly via the outbox.
2. Each re-emission gets a **fresh** Kafka envelope `eventId` (never
   reuses an old one â€” that would defeat dedupe).
3. Re-emission safety relies on the state-machine guard (Â§4), not on
   dedupe.
