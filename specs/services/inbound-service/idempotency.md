# inbound-service — Idempotency Strategy

Idempotency guarantees for `inbound-service`, covering three independent
mutation surfaces: synchronous REST, ERP webhook ingestion, and Kafka master
event consumption. This document is the single reference for idempotency
implementation in this service.

Relevant rules: trait `transactional` T1 (REST idempotency), T8 (eventId
dedupe), T2 (no distributed TX), T4 (no direct status update); trait
`integration-heavy` I4 (idempotent side-effects), I6 (webhook reception
pattern).

---

## Overview

`inbound-service` has three mutation surfaces that all require idempotency:

| Surface | Mechanism | Storage | TTL |
|---|---|---|---|
| REST (POST / PATCH) | `Idempotency-Key` header | Redis | 24h |
| ERP webhook ingest | `X-Erp-Event-Id` header | PostgreSQL `erp_webhook_dedupe` | 7 days |
| Kafka consumer (`master.*`) | `eventId` field in event envelope | PostgreSQL `inbound_event_dedupe` | 30 days |

These three mechanisms are independent. A REST call, a webhook, and a Kafka
consumer path are never deduplicated against each other — they are
fundamentally different operations with different replay characteristics.

## 1. REST Idempotency (`Idempotency-Key` header)

### 1.1 Scope

All mutating endpoints in [`specs/contracts/http/inbound-service-api.md`](../../contracts/http/inbound-service-api.md)
require `Idempotency-Key`:

| Method | Endpoints |
|---|---|
| POST | `/asns` |
| POST | `/asns/{id}:cancel`, `/asns/{id}:close` |
| POST | `/asns/{id}/inspection:start`, `/asns/{id}/inspection` |
| POST | `/inspections/{id}/discrepancies/{did}:acknowledge` |
| POST | `/asns/{id}/putaway:instruct` |
| POST | `/putaway/{iid}/lines/{lid}:confirm`, `/putaway/{iid}/lines/{lid}:skip` |

GET endpoints are naturally idempotent — no header required or accepted.
Webhook endpoint (`POST /webhooks/erp/asn`) does NOT use `Idempotency-Key` —
see §2.

### 1.2 Key Format

- Type: UUID v4 (recommended) or any opaque string ≤ 128 chars
- Generated by the **caller** (gateway client UI, integration test, ops tool)
- Must be unique per logical operation. Reuse across different logical
  operations triggers `DUPLICATE_REQUEST` (409)

### 1.3 Redis Storage Layout

Redis key pattern: `inbound:idempotency:{method}:{path_hash}:{idempotency_key}`

Where:
- `{method}`: HTTP method (`POST`, `PATCH`)
- `{path_hash}`: stable hash of the resolved request path (includes path
  variables — so `/asns/abc:cancel` and `/asns/xyz:cancel` get distinct keys)
- `{idempotency_key}`: the `Idempotency-Key` header value

Value stored: a JSON snapshot of the response (status code + body), plus a
body hash (SHA-256 of canonical request body).

TTL: 86400 seconds (24h) from first write.

> **Spec-aligned naming.** The Redis prefix MUST be `inbound:idempotency:` —
> matches the convention enforced for inventory-service (TASK-BE-025) where a
> shorter `inbound:idem:` shape was rejected as latent risk for caller-built
> keys. This is a service-level constant defined in
> `RedisIdempotencyStore.ENTRY_PREFIX`.

### 1.4 Processing Logic

On receiving a mutating request:

```
1. Read Redis entry for (method, path_hash, Idempotency-Key)

2a. Entry does NOT exist:
    → Store a PENDING marker (prevents concurrent duplicate)
    → Execute the use-case (@Transactional)
    → On success: store {statusCode, body, bodyHash} in Redis with 24h TTL
    → Return fresh response

2b. Entry exists, status = PENDING (concurrent duplicate in-flight):
    → Return 503 with Retry-After: 1

2c. Entry exists, status = COMPLETE:
    → Compare bodyHash of new request vs stored bodyHash
    → If match: return cached response (same status code + body)
    → If mismatch: return 409 DUPLICATE_REQUEST

2d. Entry exists, status = FAILED:
    → Treat as if entry does NOT exist — allow retry
    (Use-case failed previously; the caller retrying with same key is expected)
```

### 1.5 Error Responses

| Scenario | HTTP | Error Code | Notes |
|---|---|---|---|
| Header absent on POST | 400 | `VALIDATION_ERROR` | Controller guard |
| Key exists, body matches | 2xx | — | Cached response replayed |
| Key exists, body differs | 409 | `DUPLICATE_REQUEST` | Caller bug: reused key for different op |
| Key in PENDING (race) | 503 | — | `Retry-After: 1` |
| Key in FAILED | (retry fresh) | — | Prior failure was not idempotent; allow re-attempt |

### 1.6 Body Hash

Canonical body for hashing:
- Parse JSON body
- Recursively sort object keys alphabetically
- Re-serialize to compact JSON
- SHA-256 (hex string)

Purpose: detect caller sending the same key with a modified body (clock skew,
client bug, retry with different quantity). The hash comparison catches this
before a second mutation is applied.

### 1.7 Domain-Level Backstops

Two domain-level uniqueness constraints back up the Redis dedupe — they
remain effective after Redis TTL expiry:

| Endpoint | Domain backstop |
|---|---|
| `POST /asns` | `asn.asn_no` UNIQUE constraint — same `asnNo` retried after Redis eviction → `409 ASN_NO_DUPLICATE` |
| `POST /asns/{id}/inspection` | One Inspection per ASN — second create attempt sees `Asn.status != INSPECTING` → `STATE_TRANSITION_INVALID` |
| `POST /asns/{id}/putaway:instruct` | One PutawayInstruction per ASN — same as above |
| `POST /putaway/{iid}/lines/{lid}:confirm` | `PutawayLine.status` guard — second confirm sees `CONFIRMED` not `PENDING` → `STATE_TRANSITION_INVALID` |

These two-layer defenses ensure inbound idempotency survives Redis TTL
expiry. Most retries happen within 24h; the domain layer is the safety net.

## 2. Webhook Idempotency (`X-Erp-Event-Id`)

### 2.1 Scope

The single endpoint `POST /webhooks/erp/asn`. See
[`specs/contracts/webhooks/erp-asn-webhook.md`](../../contracts/webhooks/erp-asn-webhook.md)
§ Replay Dedupe for the wire-level contract.

### 2.2 ErpWebhookDedupe Table

Declared in `domain-model.md` §8. Physical table:

```sql
CREATE TABLE erp_webhook_dedupe (
  event_id     VARCHAR(80) PRIMARY KEY,
  received_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Retention: 7 days. Nightly cron deletes rows where
`received_at < now() - interval '7 days'`. 7 days covers ERP retry windows
(usually 1–24h) plus operator buffer.

### 2.3 Processing Logic — Synchronous (Controller)

Within the same `@Transactional` boundary as the inbox write:

```
1. INSERT INTO erp_webhook_dedupe (event_id, received_at)
   VALUES (?, now())
   ON CONFLICT (event_id) DO NOTHING
   RETURNING event_id

2a. Row inserted (returned):
    → INSERT INTO erp_webhook_inbox (event_id, raw_payload, signature, received_at, status)
      VALUES (?, ?, ?, now(), 'PENDING')
    → Commit TX
    → Respond 200 {status: "accepted", eventId, asnNo, receivedAt}

2b. No row returned (conflict — already exists):
    → Lookup previously stored receivedAt for the response body
    → No inbox write (already exists from prior receipt)
    → Respond 200 {status: "ignored_duplicate", eventId, previouslyReceivedAt}
```

The dedupe table is the **source of truth** for "did we already see this
event-id". The inbox is the **work queue**; in steady state, every dedupe row
has a corresponding inbox row, but the dedupe row alone is sufficient for
duplicate detection.

### 2.4 Processing Logic — Asynchronous (Background Processor)

The background processor runs every 1s, scans `erp_webhook_inbox` for
`status = PENDING` rows, and applies the domain logic. The processor is
already idempotent because:

- The inbox has the row; running `ReceiveAsnUseCase` succeeds → `status =
  APPLIED`. A future re-scan finds no `PENDING` rows for this event-id.
- If the processor crashes mid-apply: the TX rolled back. The inbox row stays
  `PENDING`. Next scan picks it up.
- If `ReceiveAsnUseCase` fails (e.g., master snapshot stale): the inbox row
  moves to `status = FAILED` with `failure_reason`. The processor does NOT
  retry FAILED rows automatically — ops triggers retry via §2.5.

### 2.5 Manual Retry of FAILED Inbox Rows

In v1: not exposed via REST. Ops investigates via SQL or admin dashboard
(read-only), and re-emission flows back through the standard ERP retry path
(after the 7-day dedupe TTL has expired, or by direct DB row deletion if the
team agrees).

In v2: `POST /api/v1/inbound/webhooks/inbox/{eventId}:retry` will reset
`status = PENDING` and the processor picks it up on next scan.

### 2.6 Webhook vs REST Crossover

`source` field on `Asn` distinguishes the two paths (`MANUAL` vs
`WEBHOOK_ERP`). They share the same `asn_no` UNIQUE constraint:

- ERP webhook arrives, creates `Asn(asn_no="ASN-X", source=WEBHOOK_ERP)`.
- An operator separately creates an ASN with the same `asnNo` via
  `POST /asns` → 409 `ASN_NO_DUPLICATE`.

So the two paths cannot accidentally double-create. The reverse (operator
creates `MANUAL` first, ERP webhook arrives later) inverts the scenario but
results in the same: the second producer hits the unique constraint.

## 3. Kafka Consumer Idempotency (EventId Dedupe)

### 3.1 Scope

All Kafka consumers in `inbound-service`. v1 only consumes `master.*`
events:

| Consumer | Topics |
|---|---|
| `MasterWarehouseConsumer` | `wms.master.warehouse.v1` |
| `MasterZoneConsumer` | `wms.master.zone.v1` |
| `MasterLocationConsumer` | `wms.master.location.v1` |
| `MasterSkuConsumer` | `wms.master.sku.v1` |
| `MasterLotConsumer` | `wms.master.lot.v1` |
| `MasterPartnerConsumer` | `wms.master.partner.v1` |

`inbound-service` does NOT consume any `inventory.*` or `outbound.*` events
in v1.

### 3.2 InboundEventDedupe Table

Declared in `domain-model.md` §6. Physical table:

```sql
CREATE TABLE inbound_event_dedupe (
  event_id     UUID        PRIMARY KEY,
  event_type   VARCHAR(60) NOT NULL,
  processed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  outcome      VARCHAR(20) NOT NULL  -- 'APPLIED' | 'IGNORED_DUPLICATE' | 'FAILED'
);
```

Retention: 30 days. A nightly cron deletes rows where `processed_at < now() -
interval '30 days'`. 30 days covers operator-initiated DLQ replays and broker
retention windows.

### 3.3 Processing Logic

Within the same `@Transactional` boundary as the snapshot upsert. The
`EventDedupePort` adapter declares `Propagation.MANDATORY` so the dedupe
write joins the consumer's TX (matches the inventory-service pattern from
TASK-BE-027):

```
1. INSERT INTO inbound_event_dedupe (event_id, event_type, processed_at, outcome)
   VALUES (?, ?, now(), 'APPLIED')
   ON CONFLICT (event_id) DO NOTHING
   RETURNING event_id

2a. Row inserted (returned):
    → Apply snapshot upsert (UPSERT on snapshot table with master_version guard)
    → Commit TX (upsert + dedupe row in same TX)

2b. No row returned (conflict — already exists):
    → Skip mutation
    → Commit TX (no-op; the IGNORED_DUPLICATE outcome is already recorded
      from the prior delivery)
    → Log: INFO "Duplicate event skipped" with {eventId, eventType, consumer}
```

### 3.4 Master Snapshot Consumer Layered Idempotency

Master snapshot upserts are doubly idempotent — the table-level UPSERT with
`master_version` guard is naturally idempotent on its own:

```sql
INSERT INTO location_snapshot (id, ..., master_version)
VALUES (?, ..., ?)
ON CONFLICT (id) DO UPDATE
  SET status = EXCLUDED.status, ..., master_version = EXCLUDED.master_version,
      cached_at = now()
  WHERE location_snapshot.master_version < EXCLUDED.master_version;
```

If `master_version <= cached`, the UPDATE clause matches no rows — naturally
idempotent.

The `inbound_event_dedupe` row is still written for two reasons:

1. **Observability**: the `outcome` field tells ops "did we just no-op a
   duplicate?" in a queryable way. The UPSERT alone doesn't surface that.
2. **Defense in depth**: future master events may carry payloads that
   require non-idempotent side effects (e.g., counter increments) where the
   dedupe table's PK is the only line of defense.

### 3.5 Retry and DLQ

Per trait `integration-heavy` rule I5:

- Spring Kafka `DefaultErrorHandler` with exponential backoff:
  - Retry 1: 1s delay
  - Retry 2: 2s delay
  - Retry 3: 4s delay
  - After 3 retries: route to `<topic>.DLT` via `DeadLetterPublishingRecoverer`
- DLT topic name pattern: `wms.master.location.v1.DLT` (Spring Kafka default
  suffix `.DLT`)
- DLT messages include original headers + `kafka_dlt-exception-*` headers
  for inspection

Permanent failures (unparseable JSON, unknown `eventType`): go to DLT
immediately (no retry). Non-retryable exception list includes
`JsonProcessingException` and custom `UnknownEventTypeException`.

Transient failures (DB connection, optimistic lock): retried per backoff
schedule.

### 3.6 Consumer Group

All consumers in `inbound-service` share consumer group ID:
`inbound-service`.

Partition assignment ensures all events for a given partition (keyed by
master entity id) are consumed by the same pod — preserving in-partition
ordering for the same entity's snapshot updates.

## 4. Decision Table — Which Mechanism Applies

| Scenario | Mechanism | Behavior |
|---|---|---|
| REST POST with same `Idempotency-Key`, same body | Redis | Replay cached 2xx |
| REST POST with same `Idempotency-Key`, different body | Redis | 409 `DUPLICATE_REQUEST` |
| REST POST `/asns`, same `asnNo`, different key, key expired | Domain unique constraint | 409 `ASN_NO_DUPLICATE` |
| REST POST `:confirm` on already-confirmed line | Domain status guard | 422 `STATE_TRANSITION_INVALID` |
| ERP webhook with same `X-Erp-Event-Id` | PostgreSQL `erp_webhook_dedupe` | 200 `ignored_duplicate`, single domain effect |
| ERP retries the same ASN with NEW `X-Erp-Event-Id` (operator deleted dedupe row by mistake) | Domain unique on `asn_no` | inbox row → status=FAILED with reason `ASN_NO_DUPLICATE` |
| Kafka broker redelivers same master event `eventId` | PostgreSQL `inbound_event_dedupe` | Skip, log as duplicate |
| Kafka DLQ replay (operator-initiated) within 30 days | PostgreSQL `inbound_event_dedupe` | Skip if already processed |
| Kafka DLQ replay after 30-day TTL purge | No dedupe row exists | Reprocess — UPSERT with version guard remains naturally idempotent for snapshots |
| Two pods consume same partition briefly (rebalance) | At-most-once in partition | Not possible — Kafka partition assigned to one consumer |

### 30-day Replay Risk for Kafka

After 30 days, the dedupe table has purged the row. An operator-initiated
replay of a master-event message older than 30 days would re-execute the
UPSERT — but the `master_version` guard means stale events are no-ops.
Practically safe.

### 7-day Replay Risk for Webhooks

After 7 days, the webhook dedupe row is purged. ERP re-emission of the same
`event_id` after that window WOULD pass the dedupe check. The result depends
on the `asn_no`:

- If the resulting ASN is identical (same `asn_no`, same content) → blocked by
  `asn.asn_no` UNIQUE constraint → inbox row marks `FAILED`. No double effect.
- If ERP re-emits with a NEW `asnNo` (e.g., they think it's a new shipment) →
  treated as a new ASN, which is correct.

The 7-day TTL is intentional: long enough for ERP retry storms to settle,
short enough to keep the dedupe table small.

## 5. Idempotency-Key Lifecycle Guidance for Callers

### REST callers (operator UI, integration tests)

1. Generate one `Idempotency-Key` (UUID v4) at the start of the user action
2. Include it on the request
3. On retry (network failure, timeout): reuse the **same key**
4. On receiving 2xx (fresh or cached): proceed
5. On receiving 409 `DUPLICATE_REQUEST`: stop — investigate the body mismatch
   (likely a UI bug)
6. On receiving 503 (PENDING): retry after the `Retry-After` interval

### ERP webhook callers

1. ERP generates one `X-Erp-Event-Id` per logical ASN event
2. On retry (5xx, timeout, no response within ERP's expected window): reuse
   the **same** `X-Erp-Event-Id`
3. On 200 `accepted`: ERP marks the event as delivered
4. On 200 `ignored_duplicate`: ERP marks delivered (we already had it)
5. On 401 (signature/timestamp): ERP investigates clock drift, secret
   rotation, integration version
6. On 422: ERP fixes the payload schema and re-emits with a NEW event-id
   (because the schema fix changes the payload — keeping the same id would
   be a body mismatch)
7. On 503: ERP retries with same id after backoff

### Operator UI for `inbound-service`

1. The ops UI generates the `Idempotency-Key` before showing the confirmation
   dialog
2. If the user double-clicks: second submission reuses the key → cached
   response returned
3. If the user leaves and returns: a new key is generated → a new
   independent operation (correct: user intentionally re-doing the action)

## 6. Observability

| Metric | Description |
|---|---|
| `inbound.idempotency.hit.rate` | % of REST POST requests served from Redis cache |
| `inbound.idempotency.mismatch.count` | `DUPLICATE_REQUEST` body mismatches (usually a bug) |
| `inbound.webhook.dedupe.hit.rate` | % of ERP webhooks served as `ignored_duplicate` |
| `inbound.webhook.inbox.pending.count` | Gauge: PENDING inbox rows; alerts at >100 |
| `inbound.webhook.inbox.failed.count` | Gauge: FAILED inbox rows; alerts at >0 |
| `inbound.event.dedupe.hit.rate` | % of consumed master events skipped as duplicate |
| `inbound.event.dedupe.table.size` | Current row count (alert if unexpectedly large) |

Log events (structured JSON, INFO level, exclude raw body and signature
value):

- `idempotency_cache_hit` `{key, method, path}` — REST replay (INFO)
- `idempotency_key_mismatch` `{key, method, path}` — body diff detected (WARN)
- `webhook_duplicate` `{eventId, source, previouslyReceivedAt}` (INFO)
- `webhook_inbox_processed` `{eventId, status, asnId|null, failureReason|null}` (INFO/WARN)
- `event_duplicate_skipped` `{eventId, eventType, consumer}` — Kafka skip (DEBUG)

## 7. Test Requirements

### Unit
- Body hash computation determinism (different key order → same hash)
- ENTRY_PREFIX value matches `inbound:idempotency:`

### Application / Slice
- REST controller: missing `Idempotency-Key` → 400 with `VALIDATION_ERROR`
- REST controller: same key + body → cached replay
- REST controller: same key + different body → 409 `DUPLICATE_REQUEST`
- Webhook controller: dup `X-Erp-Event-Id` → 200 `ignored_duplicate`, no
  inbox row written
- Background processor: re-processing a `PENDING` row that was previously
  `APPLIED` (manual SQL reset) → no second outbox row (state guard catches)

### Integration (Testcontainers)
- Redis store: literal key shape pinned with assertion (matches
  `RedisIdempotencyStoreTest` pattern from inventory-service)
- `inbound_event_dedupe` PK conflict on second delivery of same eventId
- `erp_webhook_dedupe` PK conflict on second webhook with same event-id

### Failure-mode (per trait `transactional` Required Artifact 5)
- Same `Idempotency-Key` POST twice → identical result, single mutation
- Same webhook event-id → single ASN created
- Master event redelivery → snapshot updated once, dedupe row recorded as
  `IGNORED_DUPLICATE` on second
- Redis down → REST endpoints fail closed (503 with `SERVICE_UNAVAILABLE`)
  per inventory-service convention
