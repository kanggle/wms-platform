# notification-service ‚Äî Idempotency Strategy

Detailed idempotency handling for the **consumer + delivery pipeline** of
`notification-service`. Required artifact per [`rules/traits/transactional.md`](../../../../../rules/traits/transactional.md)
rules T1 (deterministic idempotency), T3 (outbox), T8 (event dedupe).

Read after [`architecture.md`](architecture.md) and [`../../contracts/events/notification-subscriptions.md`](../../contracts/events/notification-subscriptions.md).

> **Pattern note**: This service is event-consumer + outbox publisher. Its
> idempotency surface is **two-sided**: inbound Kafka events deduped by
> `eventId`, outbound channel deliveries (Slack webhook) deduped by a
> deterministic `delivery_idempotency_key`. Contrast with
> [`master-service/idempotency.md`](../master-service/idempotency.md) which
> protects HTTP request mutations via an `Idempotency-Key` header ‚Äî same
> spirit, different surface.

---

## Scope

Two distinct idempotency boundaries:

| Boundary | Surface | Key | Store |
|---|---|---|---|
| **Inbound** | All subscribed Kafka topics (6 source topics per [`notification-subscriptions.md`](../../contracts/events/notification-subscriptions.md)) | `eventId` (UUID from envelope) | `notification_event_dedupe` table (Postgres) |
| **Outbound** | All channel dispatches (Slack webhook v1) | `delivery_idempotency_key = sha256(eventId + channelId + recipient)` | `notification_delivery` table column `delivery_idempotency_key UQ` |

The service publishes one event itself (`wms.notification.delivered.v1`)
via outbox; its envelope carries an `eventId` generated by
[`libs/java-messaging`](../../../../../libs/java-messaging/) ‚Äî downstream
consumers (currently `admin-service` only) own their inbound dedupe.

## Inbound Idempotency (Kafka)

### Table

```
notification_event_dedupe (
    event_id        VARCHAR(36) PRIMARY KEY,    -- source envelope's eventId (UUID)
    source_topic    VARCHAR(128) NOT NULL,      -- e.g. "wms.inventory.alert.v1"
    processed_at    TIMESTAMPTZ  NOT NULL,
    outcome         VARCHAR(16)  NOT NULL,      -- QUEUED | FILTERED
    constraint chk_outcome check (outcome in ('QUEUED','FILTERED'))
)
```

- **PK = `event_id` alone.** The same `eventId` MUST be unique across the
  service; reusing it for two different topics is treated as a duplicate
  (covers the rare cross-topic replay case).
- **`source_topic`** preserved for diagnostics + DLT replay
  ([`runbooks/dlt-replay.md`](runbooks/dlt-replay.md)).
- **`outcome`** records why the event was (or was not) actioned:
  - `QUEUED` ‚Äî routing rule matched and `NotificationDelivery` rows persisted.
  - `FILTERED` ‚Äî routing predicate excluded the event (e.g.
    `|delta| < 100` for an `inventory.adjusted` event). Records the
    classification so replay produces the same answer.

### Retention

**30 days.** A nightly job in `inventory-service`-style `@Scheduled`
manner is **not v1**; rely on a one-shot pg admin migration when the
table grows beyond ~5 GB. Operators can `DELETE FROM
notification_event_dedupe WHERE processed_at < now() - interval '30 days'`
during a maintenance window.

### Lookup + Insert flow

Inside the application service, in **one DB transaction**:

1. `SELECT 1 FROM notification_event_dedupe WHERE event_id = ?`
   - **Hit** ‚Üí exit cleanly, do not produce deliveries, do not append
     outbox. Caller acks the Kafka offset.
   - **Miss** ‚Üí continue.
2. Resolve routing rule by `eventType` (cached in-memory, 60s TTL ‚Äî
   `architecture.md` ¬ß Concurrency Control).
3. Evaluate matcher predicate.
   - **No match** ‚Üí INSERT row with `outcome = FILTERED`. COMMIT. Ack.
   - **Match** ‚Üí continue.
4. Build `NotificationDelivery` rows per channel target (status =
   `PENDING`, `attempt_count = 0`, `delivery_idempotency_key` computed
   below).
5. INSERT dedupe row with `outcome = QUEUED`, INSERT delivery rows,
   APPEND outbox `wms.notification.delivered.v1` row. **All in one
   transaction.** COMMIT.

### Replay semantics

- Same `eventId` second time ‚Üí step 1 hit ‚Üí no side effects. Behavior
  is identical regardless of original `outcome` (`QUEUED` or `FILTERED`).
- Cross-context replay (e.g. service restart, consumer rebalance) ‚Üí same
  guarantee ‚Äî eventId-based dedupe wins.

## Outbound Idempotency (Channel)

### Key construction

```
delivery_idempotency_key = sha256(
    eventId        ||      -- envelope's eventId
    ":" ||
    channelId      ||      -- "slack:#wms-alerts" (canonical channel id)
    ":" ||
    recipient              -- channel-specific target ("@oncall" / "" for default)
)
```

Hex-encoded SHA-256, stored in `notification_delivery.delivery_idempotency_key`
column (`VARCHAR(64) UNIQUE NOT NULL`).

### Per-row attempt accounting

- `NotificationDelivery` row created with `attempt_count = 0`,
  `status = PENDING`.
- Each Slack webhook call increments `attempt_count` **in place** (same
  row). UNIQUE on `delivery_idempotency_key` guarantees no second row.
- Vendor-side double-fire is **possible** (Slack does not enforce
  idempotency on Incoming Webhooks). The key persists across retries so
  audit trail is consistent even when Slack itself accepts the same
  message twice.

### Retry budget

| Field | Default | Bound |
|---|---|---|
| `max_attempts` | **5** | hard cap, set in routing rule per channel |
| `attempt_count` | 0..5 | incremented on each dispatch attempt |
| backoff | exponential + jitter: 30s, 60s, 2m, 5m, 10m | `scheduled_retry_at` column |
| terminal outcome | `SUCCEEDED` or `FAILED` (`attempt_count >= max_attempts`) | enforced by `PostStatusMachine`-style guard |

On `attempt_count >= max_attempts` AND last attempt fails:
- `status` ‚Üí `FAILED`
- Outbox emits terminal audit event `wms.notification.delivery.failed.v1`
  with `outcome = "FAILED_RETRY_EXHAUSTED"` and error code
  `DELIVERY_RETRY_EXHAUSTED` (per [`platform/error-handling.md`](../../../../../platform/error-handling.md)).
- Operator surface: see [`runbooks/dlt-replay.md`](runbooks/dlt-replay.md)
  for manual replay procedure.

This is the ADR-MONO-005 **Category C** pattern (single-step retry +
DLT) ‚Äî see [`docs/adr/`](../../../../../docs/adr/) ¬ß ADR-MONO-005 ¬ß D5
Category C.

## Concurrency Control

- `NotificationDelivery` carries `version` (JPA `@Version` optimistic
  lock). Two retry workers picking the same row simultaneously ‚Üí second
  call's commit fails on version mismatch ‚Üí falls through to next tick.
- The retry scheduler reads rows with `SELECT ‚Ä¶ FOR UPDATE SKIP LOCKED`
  so two worker instances cannot double-fire the same delivery. The
  scheduler is a stock Spring `@Scheduled` running every 30 seconds.
- Routing rules are read-mostly; an in-memory cache with a 60s TTL is
  acceptable in v1 (operator config edit propagates after one minute).
- The dedupe table itself does not need explicit locking ‚Äî PK insert
  raises `DataIntegrityViolationException` on conflict and the consumer
  treats it as a hit.

## Failure Modes

| Failure | Behavior |
|---|---|
| Postgres unreachable during dedupe lookup | **Fail closed**: consumer throws, Kafka offset not committed, message redelivered by next poll. T1 is mandatory. |
| Postgres reachable but `notification_event_dedupe` constraint violation (race between two consumer instances on same `eventId`) | Treat the `DataIntegrityViolationException` as a hit ‚Äî second caller exits cleanly without side effects. |
| Slack webhook 5xx / timeout | Retry per ¬ß Retry budget. `attempt_count++`, `scheduled_retry_at` set, next scheduler tick picks up. |
| Slack webhook 4xx (malformed payload) | Terminal: do NOT retry. `status = FAILED`, `last_error = "<4xx body>"`, outbox terminal event. Surfaces as production bug ‚Äî see [`runbooks/dlt-replay.md`](runbooks/dlt-replay.md). |
| Slack webhook 200 but message not visible (vendor delivery failure) | Out of our visibility surface ‚Äî we account success at HTTP layer only. Per `architecture.md` ¬ß Channel adapter design. |
| Outbox publish failure (Postgres CDC lag, etc.) | Inherits libs/java-messaging behavior ‚Äî `outbox_events.published_at IS NULL` until polled; replay-safe. |
| Spring Kafka DLT routing (3 retries with exponential backoff exhausted) | Message lands on `<source-topic>.DLT` (per `KafkaConsumerConfig`'s `DefaultErrorHandler`). Operator drains via [`runbooks/dlt-replay.md`](runbooks/dlt-replay.md). |
| Cross-context replay (service restart mid-delivery) | Inbound dedupe wins; outbound `delivery_idempotency_key` UQ prevents row duplication; retry scheduler picks up `PENDING` rows naturally. |

## Observability

Metrics (Micrometer):

| Metric | Tags | Meaning |
|---|---|---|
| `wms.notification.dedupe.outcome` | `outcome={hit,fresh-queued,fresh-filtered}`, `source_topic` | Inbound dedupe classification at consumer entry |
| `wms.notification.delivery.attempt` | `channel`, `outcome={success,retry,terminal}` | Per dispatch attempt result |
| `wms.notification.delivery.terminal` | `channel`, `reason={SUCCEEDED,FAILED_RETRY_EXHAUSTED,FAILED_4XX}` | Counted when row reaches terminal state |
| `wms.notification.dedupe.constraint.violation` | `source_topic` | Race-loss count (informational, expected at low rate) |
| `wms.notification.delivery.retry.queue.depth` | ‚Äî | Gauge: count of rows where `status=PENDING AND attempt_count > 0` |

Logs:

- On dedupe hit: log at `DEBUG` with `eventId`, `source_topic`. Never log the entire envelope payload.
- On `FAILED_RETRY_EXHAUSTED`: log at `WARN` with `eventId`, `channelId`, `last_error`, `attempt_count`.
- On terminal 4xx: log at `ERROR` with the truncated response body (no PII).

Tracing:

- Inbound consumer span carries `messaging.kafka.partition.id` + `eventId`.
- Each dispatch attempt carries `notification.attempt = N` span attribute.
- Replay (hit) span carries `notification.dedupe.hit = true`.

## Testing Requirements

Every change to dedupe / delivery / retry logic MUST cover:

1. **First event** ‚Äî fresh dedupe row + delivery row + outbox row (one TX).
2. **Replay (same eventId)** ‚Äî no second delivery row, no second outbox row, dedupe row unchanged.
3. **Filtered path** ‚Äî predicate fails ‚Üí `FILTERED` dedupe row + zero deliveries + zero outbox rows.
4. **Retry budget exhausted** ‚Äî 5 failed attempts ‚Üí terminal `FAILED`, outbox terminal event emitted, `DELIVERY_RETRY_EXHAUSTED` error code surfaced.
5. **Slack 4xx** ‚Äî single attempt ‚Üí terminal `FAILED`, no retry.
6. **Postgres outage during dedupe** ‚Äî Testcontainers pause ‚Üí consumer throws ‚Üí offset uncommitted ‚Üí resumes on poll after pause release.
7. **Concurrent same-eventId (two consumer instances)** ‚Äî one wins, the other observes the constraint violation and exits cleanly.
8. **Cross-context replay (full Spring context restart)** ‚Äî same eventId, same envelope ‚Üí dedupe hit ‚Üí no duplicate Slack call.

Reference: [`platform/testing-strategy.md`](../../../../../platform/testing-strategy.md) + sibling [`master-service/idempotency.md`](../master-service/idempotency.md) ¬ß Testing Requirements (Testcontainers Postgres patterns are reusable).

## Not In v1

- **Dedupe table sweeper job** ‚Äî manual delete during maintenance.
- **Per-tenant dedupe scoping** ‚Äî single-tenant in v1; if multi-tenant introduced later, add `tenant_id` column to PK.
- **Cross-service dedupe sharing** ‚Äî each service owns its own dedupe table.
- **Replay UI / admin API** ‚Äî DLT replay is via [`runbooks/dlt-replay.md`](runbooks/dlt-replay.md) operator playbook; no first-class API yet.
- **Idempotency for outbound publishes** (`wms.notification.delivered.v1`) ‚Äî inherits `libs/java-messaging`'s `BaseEventPublisher` outbox guarantee; downstream consumers (`admin-service`) own dedupe.
