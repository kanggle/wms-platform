# inventory-service ŌĆö Idempotency Strategy

Idempotency guarantees for `inventory-service`, covering both the synchronous REST
surface and the asynchronous Kafka consumer surface. This document is the single
reference for idempotency implementation in this service.

Relevant rules: trait `transactional` T1 (REST idempotency), T8 (eventId dedupe),
T2 (no distributed TX), T4 (no direct status update).

---

## Overview

`inventory-service` has two mutation surfaces that both require idempotency:

| Surface | Mechanism | Storage | TTL |
|---|---|---|---|
| REST (POST / PATCH) | `Idempotency-Key` header | Redis | 24h |
| Kafka consumer | `eventId` field in event envelope | PostgreSQL | 30 days |

These two mechanisms are independent. A REST call and a Kafka consumer path are
never deduplicated against each other ŌĆö they are fundamentally different operations.

---

## 1. REST Idempotency (`Idempotency-Key` header)

### 1.1 Scope

All mutating endpoints require `Idempotency-Key`:

| Method | Endpoints |
|---|---|
| POST | `/adjustments`, `/{id}/mark-damaged`, `/{id}/write-off-damaged` |
| POST | `/transfers` |
| POST | `/reservations`, `/reservations/{id}/confirm`, `/reservations/{id}/release` |

GET endpoints are naturally idempotent ŌĆö no header required or accepted.

### 1.2 Key Format

- Type: UUID v4 (recommended) or any opaque string Ōēż 128 chars
- Generated by the **caller** (gateway, `outbound-service`, ops UI)
- Must be unique per logical operation. Reuse across different logical operations
  (e.g., the same UUID sent for two different reservation creates) triggers `DUPLICATE_REQUEST` (409)

### 1.3 Redis Storage Layout

Redis key pattern: `inventory:idempotency:{method}:{path_hash}:{idempotency_key}`

Where:
- `{method}`: HTTP method (`POST`, `PATCH`)
- `{path_hash}`: stable hash of the resolved request path (includes path variables)
- `{idempotency_key}`: the `Idempotency-Key` header value

Value stored: a JSON snapshot of the response (status code + body), plus a body
hash (SHA-256 of canonical request body).

TTL: 86400 seconds (24h) from first write.

### 1.4 Processing Logic

On receiving a mutating request:

```
1. Read Redis entry for (method, path_hash, Idempotency-Key)

2a. Entry does NOT exist:
    ŌåÆ Store a PENDING marker (prevents concurrent duplicate)
    ŌåÆ Execute the use-case (@Transactional)
    ŌåÆ On success: store {statusCode, body, bodyHash} in Redis with 24h TTL
    ŌåÆ Return fresh response

2b. Entry exists, status = PENDING (concurrent duplicate in-flight):
    ŌåÆ Return 503 with Retry-After: 1

2c. Entry exists, status = COMPLETE:
    ŌåÆ Compare bodyHash of new request vs stored bodyHash
    ŌåÆ If match: return cached response (same status code + body)
    ŌåÆ If mismatch: return 409 DUPLICATE_REQUEST

2d. Entry exists, status = FAILED:
    ŌåÆ Treat as if entry does NOT exist ŌĆö allow retry
    (Use-case failed previously; the caller retrying with same key is expected)
```

### 1.5 Error Responses

| Scenario | HTTP | Error Code | Notes |
|---|---|---|---|
| Header absent on POST | 400 | `VALIDATION_ERROR` | Controller guard |
| Key exists, body matches | 2xx | ŌĆö | Cached response replayed |
| Key exists, body differs | 409 | `DUPLICATE_REQUEST` | Caller bug: reused key for different op |
| Key in PENDING (race) | 503 | ŌĆö | `Retry-After: 1` |
| Key in FAILED | (retry fresh) | ŌĆö | Prior failure was not idempotent; allow re-attempt |

### 1.6 Body Hash

Canonical body for hashing:
- Parse JSON body
- Recursively sort object keys alphabetically
- Re-serialize to compact JSON
- SHA-256 (hex string)

Purpose: detect caller sending the same key with a modified body (clock skew, client
bug, retry with different quantity). The hash comparison catches this before a second
mutation is applied.

### 1.7 Domain-Level Idempotency for Reservations

`Reservation.picking_request_id` enforces a stronger guarantee at the domain level:

- A reservation for a given `pickingRequestId` can only exist once (unique constraint).
- If `POST /reservations` is called twice with the same `pickingRequestId` and same
  `Idempotency-Key`: Redis dedupe catches it ŌåÆ cached 201 returned.
- If the same `pickingRequestId` is sent with a **different** `Idempotency-Key`
  (e.g., after a Redis eviction): the use-case attempts to insert, hits the unique
  constraint on `picking_request_id`, and returns 409 `DUPLICATE_REQUEST`.
- This two-layer defense ensures reservation idempotency survives Redis TTL expiry
  (most Redis failures happen within 24h; but domain-layer check is the safety net).

## 2. Kafka Consumer Idempotency (EventId Dedupe)

### 2.1 Scope

All Kafka consumers in `inventory-service`:

| Consumer | Topics |
|---|---|
| `PutawayCompletedConsumer` | `wms.inbound.putaway.completed.v1` |
| `PickingRequestedConsumer` | `wms.outbound.picking.requested.v1` |
| `PickingCancelledConsumer` | `wms.outbound.picking.cancelled.v1` |
| `ShippingConfirmedConsumer` | `wms.outbound.shipping.confirmed.v1` |
| `MasterLocationConsumer` | `wms.master.location.v1` |
| `MasterSkuConsumer` | `wms.master.sku.v1` |
| `MasterLotConsumer` | `wms.master.lot.v1` |

### 2.2 EventDedupe Table

Declared in `domain-model.md` ┬¦7. Physical table: `inventory_event_dedupe`.

```sql
CREATE TABLE inventory_event_dedupe (
  event_id    UUID        PRIMARY KEY,
  event_type  VARCHAR(60) NOT NULL,
  processed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  outcome     VARCHAR(20) NOT NULL  -- 'APPLIED' | 'IGNORED_DUPLICATE' | 'FAILED'
);
```

Retention: 30 days. A nightly cron deletes rows where `processed_at < now() - interval '30 days'`.
30 days covers operator-initiated DLQ replays and broker retention windows.

### 2.3 Processing Logic

Within the same `@Transactional` boundary as the domain mutation:

```
1. INSERT INTO inventory_event_dedupe (event_id, event_type, processed_at, outcome)
   VALUES (?, ?, now(), 'APPLIED')
   ON CONFLICT (event_id) DO NOTHING
   RETURNING id

2a. Row inserted (returned):
    ŌåÆ Apply domain mutation (receive, reserve, release, confirm, master upsert)
    ŌåÆ Commit TX (mutation + dedupe row in same TX)

2b. No row returned (conflict ŌĆö already exists):
    ŌåÆ Skip mutation
    ŌåÆ Commit TX (no-op; the IGNORED_DUPLICATE outcome is already recorded)
    ŌåÆ Log: INFO "Duplicate event skipped" with {eventId, eventType, consumer}
```

Because the dedupe insert and the domain mutation are in the same transaction:
- If the TX commits: mutation applied, `outcome = APPLIED`
- If the TX rolls back: dedupe row is also rolled back ŌåÆ next delivery retries fresh

This is the standard at-least-once + dedupe-table pattern (T8). It is safe against
broker redelivery, consumer group rebalance, and operator-initiated DLQ replay.

### 2.4 Master Snapshot Consumer (special case)

`MasterLocationConsumer`, `MasterSkuConsumer`, `MasterLotConsumer` perform UPSERT
operations on snapshot tables. These are already naturally idempotent (upsert by id
with version guard). The dedupe table is still used to prevent double-counting
in observability metrics (`outcome = IGNORED_DUPLICATE` is tracked).

The version guard:

```sql
INSERT INTO location_snapshot (..., master_version)
VALUES (?, ?, ..., ?)
ON CONFLICT (id) DO UPDATE
  SET status = EXCLUDED.status, ..., master_version = EXCLUDED.master_version, cached_at = now()
  WHERE location_snapshot.master_version < EXCLUDED.master_version;
```

If `master_version <= cached`, the UPDATE clause matches no rows ŌĆö idempotent without
the dedupe table. The dedupe table is supplementary here.

### 2.5 Retry and DLQ

Per trait `integration-heavy` rule I5:

- Spring Kafka `DefaultErrorHandler` with exponential backoff:
  - Retry 1: 1s delay
  - Retry 2: 2s delay
  - Retry 3: 4s delay
  - After 3 retries: route to `<topic>.DLT` via `DeadLetterPublishingRecoverer`
- DLT topic name pattern: `wms.inventory.received.v1.DLT` (Spring Kafka default suffix `.DLT`)
- DLT messages include original headers + `kafka_dlt-exception-*` headers for inspection

Permanent failures (unparseable JSON, unknown `eventType`): go to DLT immediately
(no retry). `SeekToCurrentErrorHandler` non-retryable exception list includes
`JsonProcessingException` and custom `UnknownEventTypeException`.

Transient failures (DB connection, optimistic lock): retried per backoff schedule.
An optimistic lock exception in a consumer context is a transient failure ŌĆö the
Spring Kafka retry will re-deliver the message and the use-case re-reads fresh state.

### 2.6 Consumer Group

All consumers in `inventory-service` share consumer group ID: `inventory-service`.

Partition assignment ensures all events for a given partition (keyed by `locationId`)
are consumed by the same pod ŌĆö preserving in-partition ordering for the same location's
stock changes.

## 3. Decision Table ŌĆö Which Mechanism Applies

| Scenario | Mechanism | Behavior |
|---|---|---|
| REST POST with same `Idempotency-Key`, same body | Redis | Replay cached 2xx |
| REST POST with same `Idempotency-Key`, different body | Redis | 409 `DUPLICATE_REQUEST` |
| REST POST, same `pickingRequestId`, different key, key expired | Domain unique constraint | 409 `DUPLICATE_REQUEST` |
| Kafka broker redelivers same `eventId` | PostgreSQL dedupe table | Skip, log as duplicate |
| Kafka DLQ replay (operator-initiated) | PostgreSQL dedupe table | Skip if already processed |
| Kafka DLQ replay after 30-day TTL purge | No dedupe row exists | Reprocess ŌĆö accepted risk |
| Two pods consume same partition briefly (rebalance) | At-most-once in partition | Not possible ŌĆö Kafka partition assigned to one consumer |

### 30-day Replay Risk

After 30 days, the dedupe table has purged the row. An operator-initiated replay of
a message older than 30 days would be re-applied. This is an accepted risk in v1:

- Broker retention is minimum 7 days ŌĆö replaying older-than-30-day messages is an
  unusual ops scenario
- If it occurs, the mutation is applied again (double receive, double reserve, etc.)
- Mitigation: operators replaying DLQ messages older than 30 days must verify the
  current inventory state first and use the manual adjustment flow if needed

## 4. Idempotency-Key Lifecycle Guidance for Callers

For `outbound-service` calling `POST /reservations`:

1. Generate one `Idempotency-Key` (UUID) at the start of the saga step
2. Store the key alongside the `OutboundSaga` row (or in the Kafka message that triggers it)
3. On retry (saga sweeper or manual re-trigger), reuse the **same key**
4. On receiving a 2xx (fresh or cached): saga advances
5. On receiving 409 `DUPLICATE_REQUEST`: stop ŌĆö investigate the body mismatch (bug)
6. On receiving 503 (PENDING): retry after the `Retry-After` interval

For human ops calling `POST /adjustments`:

1. The ops UI generates the `Idempotency-Key` before showing the confirmation dialog
2. If the user double-clicks: second submission reuses the key ŌåÆ cached response returned
3. If the user leaves and returns: the key was ephemeral in the UI ŌåÆ a new key
   generates a new adjustment (correct behavior ŌĆö user intentionally re-adjusting)

## 5. Observability

| Metric | Description |
|---|---|
| `inventory.idempotency.hit.rate` | % of POST requests served from Redis cache |
| `inventory.idempotency.mismatch.count` | `DUPLICATE_REQUEST` body mismatches (usually a bug) |
| `inventory.event.dedupe.hit.rate` | % of consumed events skipped as duplicate |
| `inventory.event.dedupe.table.size` | Current row count (alert if unexpectedly large) |

Log events (structured JSON, INFO level):

- `"idempotency_cache_hit"` with `{key, method, path}` ŌĆö REST replay
- `"event_duplicate_skipped"` with `{eventId, eventType, consumer}` ŌĆö Kafka skip
- `"idempotency_key_mismatch"` with `{key, method, path}` ŌĆö body diff detected (WARN)
