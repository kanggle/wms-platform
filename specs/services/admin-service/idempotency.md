# admin-service — Idempotency Strategy

Idempotency guarantees for `admin-service`, covering both the synchronous REST
surface and the asynchronous Kafka projection-consumer surface. This document
is the single reference for idempotency implementation in this service.

Relevant rules: trait `transactional` T1 (REST idempotency), T3 (outbox), T8
(eventId dedupe). `admin-service` does not participate in distributed
transactions (T2 trivially holds — write side does not span aggregates except
for the documented force-deactivation cascade, which is a single-DB
transaction).

---

## Overview

`admin-service` has two mutation surfaces with different idempotency needs:

| Surface | Mechanism | Storage | TTL |
|---|---|---|---|
| REST (POST / PATCH / PUT / DELETE on `/users` / `/roles` / `/assignments` / `/settings` / `/dashboard/alerts/*/acknowledge`) | `Idempotency-Key` header | Redis | 24h |
| Kafka projection consumers | `eventId` field in event envelope | PostgreSQL (`admin_event_dedupe`) + `last_event_at` last-write-wins | 30 days |

The two mechanisms are independent. A REST call and a Kafka projection are
different operations and never deduplicated against each other.

Read endpoints (`GET /api/v1/admin/dashboard/**` and similar) are pure reads —
no idempotency required.

---

## 1. REST Idempotency (`Idempotency-Key` header)

### 1.1 Scope

All mutating endpoints require `Idempotency-Key`:

| Method | Endpoint |
|---|---|
| POST | `/api/v1/admin/users` |
| PATCH | `/api/v1/admin/users/{id}` |
| POST | `/api/v1/admin/users/{id}/deactivate` |
| POST | `/api/v1/admin/users/{id}/reactivate` |
| POST | `/api/v1/admin/roles` |
| PATCH | `/api/v1/admin/roles/{id}` |
| POST | `/api/v1/admin/roles/{id}/deactivate` |
| POST | `/api/v1/admin/roles/{id}/reactivate` |
| POST | `/api/v1/admin/assignments` |
| DELETE | `/api/v1/admin/assignments/{id}` |
| PUT | `/api/v1/admin/settings/{key}` |
| POST | `/api/v1/admin/dashboard/alerts/{alertId}/acknowledge` |

GET endpoints are naturally idempotent — header not required and not accepted
(silently ignored).

### 1.2 Key Format

- Type: UUID v4 (recommended) or any opaque string ≤ 128 chars
- Generated by the **caller** (admin UI, ops scripts)
- Must be unique per logical operation. Reuse across different logical
  operations triggers `DUPLICATE_REQUEST` (409).

### 1.3 Redis Storage Layout

Redis key pattern: `admin:idempotency:{method}:{path_hash}:{idempotency_key}`

Where:
- `{method}`: HTTP method (`POST`, `PATCH`, `PUT`, `DELETE`)
- `{path_hash}`: stable hash (SHA-256 prefix) of the resolved request path
  including path variables (e.g., the user id is part of the path hash for
  `PATCH /users/{id}`)
- `{idempotency_key}`: the `Idempotency-Key` header value

Value stored: a JSON snapshot `{ status, headers, body, bodyHash, state }`.

`state` ∈ `PENDING | COMPLETE | FAILED`.

TTL: 86400 seconds (24h) from first write. The TTL is set on the PENDING
marker and refreshed at COMPLETE — a request that takes 30 minutes still gets
the full 24h cache from completion time.

### 1.4 Processing Logic

On receiving a mutating request:

```
1. Compute (method, path_hash, Idempotency-Key)

2. Read Redis entry

3a. Entry does NOT exist:
    → SET key to {state: PENDING, bodyHash: <sha256-of-canonical-body>}
       with NX (atomic create)
    → Execute the use-case (@Transactional)
    → On success: UPDATE key to {state: COMPLETE, status, headers, body, bodyHash}
       and refresh 24h TTL
    → Return fresh response

3b. Entry exists, state = PENDING:
    → Return 503 SERVICE_UNAVAILABLE with Retry-After: 1
    (concurrent duplicate in flight)

3c. Entry exists, state = COMPLETE:
    → Compute body hash of new request
    → If match: return cached response (status + headers + body)
    → If mismatch: return 409 DUPLICATE_REQUEST

3d. Entry exists, state = FAILED:
    → Treat as if entry does NOT exist — allow retry
    (Use-case failed previously; the caller retrying with same key is expected)
```

### 1.5 Error Responses

| Scenario | HTTP | Error code | Notes |
|---|---|---|---|
| Header absent on mutating endpoint | 400 | `VALIDATION_ERROR` | Filter / controller guard |
| Key exists, body matches | 2xx | — | Cached response replayed |
| Key exists, body differs | 409 | `DUPLICATE_REQUEST` | Caller bug: reused key for different op |
| Key in PENDING (race) | 503 | — | `Retry-After: 1` |
| Key in FAILED | (retry fresh) | — | Prior failure was not idempotent; allow re-attempt |

### 1.6 Body Hash

Canonical body for hashing:

- Parse JSON body
- Recursively sort object keys alphabetically
- Re-serialize to compact JSON
- SHA-256 (hex string)

Endpoints with empty body (e.g., `POST /reactivate`, `DELETE /assignments/{id}`,
`POST /alerts/{id}/acknowledge`) hash the empty string `""` — duplicates with
the same key collapse trivially.

### 1.7 Domain-Level Idempotency Safety Nets

Some surfaces have a stronger natural guard than Redis:

- **Assignment grant**: `(user_id, role_id, warehouse_id)` is unique among
  `ACTIVE` assignments. A duplicate `POST /assignments` with a fresh
  `Idempotency-Key` (after 24h Redis eviction) returns the existing active
  row with HTTP `200` rather than creating a duplicate. Per
  [`admin-service-api.md § 4.1`](../../contracts/http/admin-service-api.md).

- **User email**: globally unique constraint catches duplicate inserts even
  if the Redis cache evicted. Returns `USER_EMAIL_DUPLICATE` (409).

- **Role code**: globally unique constraint. Returns `ROLE_CODE_DUPLICATE` (409).

- **Setting key**: PK constraint on `(key, warehouse_id)`. `PUT` is upsert;
  duplicate-with-same-value is naturally idempotent at the DB level (the
  version-checked update only fires the outbox event when `value_json`
  actually changes).

- **Alert acknowledge**: `acknowledged_at` is set only when null. A second
  acknowledge attempt returns `STATE_TRANSITION_INVALID` (422) — caller
  should treat this as a successful no-op.

## 2. Kafka Consumer Idempotency (EventId Dedupe + Last-Write-Wins)

### 2.1 Scope

All projection consumers in `admin-service` (consumer group
`admin-projection`):

| Consumer | Topics |
|---|---|
| `MasterRefProjectionConsumer` | `wms.master.warehouse.v1`, `wms.master.zone.v1`, `wms.master.location.v1`, `wms.master.sku.v1`, `wms.master.partner.v1`, `wms.master.lot.v1` |
| `InboundProjectionConsumer` | `wms.inbound.asn.v1`, `wms.inbound.inspection.completed.v1`, `wms.inbound.putaway.completed.v1` |
| `OutboundProjectionConsumer` | `wms.outbound.order.v1`, `wms.outbound.shipping.confirmed.v1` |
| `InventoryProjectionConsumer` | `wms.inventory.received.v1`, `wms.inventory.adjusted.v1`, `wms.inventory.transferred.v1`, `wms.inventory.reserved.v1`, `wms.inventory.released.v1`, `wms.inventory.confirmed.v1`, `wms.inventory.alert.v1` |

The grouping is implementation-flavor — one Kafka listener class per source
service for clarity. Each listener routes by `eventType` to the right
projection method internally.

### 2.2 EventDedupe Table

Declared in `domain-model.md` § 15. Physical table: `admin_event_dedupe`.

```sql
CREATE TABLE admin_event_dedupe (
  event_id     UUID         PRIMARY KEY,
  event_type   VARCHAR(60)  NOT NULL,
  processed_at TIMESTAMPTZ  NOT NULL DEFAULT now(),
  outcome      VARCHAR(30)  NOT NULL  -- 'APPLIED' | 'IGNORED_DUPLICATE' | 'IGNORED_DUPLICATE_LATE' | 'FAILED'
);

CREATE INDEX admin_event_dedupe_processed_at_idx
  ON admin_event_dedupe (processed_at);
```

`IGNORED_DUPLICATE` covers redelivery of the same `eventId`.
`IGNORED_DUPLICATE_LATE` covers a fresh `eventId` whose `occurredAt` is older
than the persisted `last_event_at` on the target row — the dedupe row is
inserted but the projection mutation is skipped.

Retention: 30 days. A nightly cron deletes
`processed_at < now() - interval '30 days'`. 30 days covers operator-initiated
DLQ replays and broker retention windows.

### 2.3 Processing Logic

Within the same `@Transactional` boundary as the projection mutation:

```
1. INSERT INTO admin_event_dedupe (event_id, event_type, processed_at, outcome)
   VALUES (?, ?, now(), 'APPLIED')
   ON CONFLICT (event_id) DO NOTHING
   RETURNING event_id;

2a. Row inserted (new event):
    → Look up the affected read-model row(s)
    → If event.occurredAt > row.last_event_at (or row does not yet exist):
        → Apply mutation (upsert / append / increment)
        → Set row.last_event_at = event.occurredAt
        → Commit TX (mutation + dedupe row, both with outcome=APPLIED)
    → Else (out-of-order arrival):
        → UPDATE admin_event_dedupe SET outcome = 'IGNORED_DUPLICATE_LATE'
           WHERE event_id = ?
        → Commit TX (no projection mutation)
        → Log: INFO "Stale event dropped" with {eventId, eventType, eventOccurredAt, rowLastEventAt}

2b. No row returned (eventId conflict — already processed):
    → Skip mutation
    → Commit TX (no-op; dedupe row already records prior outcome)
    → Log: INFO "Duplicate event skipped" with {eventId, eventType, consumer}
    → Increment metric admin.event.dedupe.hit.count
```

The dedupe insert and the projection mutation share the transaction. If the
TX rolls back (e.g., transient DB error), the dedupe row is also rolled back
and the next delivery retries fresh. This is the standard at-least-once +
dedupe-table pattern (T8).

### 2.4 Append-Only Tables (`adjustment_audit`, `alert_log`)

For append-only tables the projection inserts a new row keyed by `eventId`
(the row PK = the source event's UUIDv7). The PK constraint provides an
additional safety net beyond the dedupe table:

```sql
INSERT INTO admin_adjustment_audit (id, ...)
VALUES (?, ...)
ON CONFLICT (id) DO NOTHING;
```

Even if the dedupe table is somehow wiped (e.g., manual TRUNCATE during
development), the PK conflict prevents double-append. Last-write-wins does
not apply — these tables grow monotonically.

### 2.5 Upsert Tables (`*_ref`, `*_summary`, `inventory_snapshot`,
`throughput_*`)

For upsert tables, the projection runs a version-checked `UPDATE`:

```sql
UPDATE admin_inventory_snapshot
SET available_qty = ?, reserved_qty = ?, ..., last_event_at = ?, version = version + 1
WHERE location_id = ? AND sku_id = ? AND lot_id IS NOT DISTINCT FROM ?
  AND last_event_at < ?;  -- last-write-wins guard
```

If 0 rows matched, the row was either absent (insert path) or its
`last_event_at` is already newer (skip path). The dedupe row reflects the
outcome accordingly.

### 2.6 Throughput Counters (Atomic Increment)

`throughput_inbound_daily` and `throughput_outbound_daily` use PostgreSQL
upsert-with-increment:

```sql
INSERT INTO admin_throughput_inbound_daily (
  date, warehouse_id, putaway_count, qty_received, last_event_at, version
)
VALUES (?, ?, 1, ?, ?, 1)
ON CONFLICT (date, warehouse_id) DO UPDATE
  SET putaway_count = admin_throughput_inbound_daily.putaway_count + 1,
      qty_received = admin_throughput_inbound_daily.qty_received + EXCLUDED.qty_received,
      last_event_at = EXCLUDED.last_event_at,
      version = admin_throughput_inbound_daily.version + 1
  WHERE admin_throughput_inbound_daily.last_event_at < EXCLUDED.last_event_at;
```

EventDedupe ensures each event increments exactly once. The
`last_event_at` guard prevents out-of-order events from incorrectly
incrementing counters using a stale source.

### 2.7 Retry and DLT

Per trait `integration-heavy` rule I5:

- Spring Kafka `DefaultErrorHandler` with exponential backoff:
  - Retry 1: 1s delay
  - Retry 2: 2s delay
  - Retry 3: 4s delay
  - After 3 retries → DLT via `DeadLetterPublishingRecoverer`
- DLT topic name pattern: `<source-topic>.DLT` (Spring Kafka default suffix)
- DLT messages include original headers + `kafka_dlt-exception-*` headers

Permanent failures (unparseable JSON, unknown `eventType`, unknown enum
value): route to DLT immediately. The non-retryable-exception list includes
`JsonProcessingException`, `IllegalArgumentException` (from enum parsing),
and a custom `UnknownEventTypeException`.

Transient failures (DB connection, optimistic lock, deadlock): retried per
backoff schedule. An optimistic lock exception in a projection context is a
transient failure — retry re-reads fresh state and re-applies.

### 2.8 Consumer Group

All consumers share `admin-projection`. Partition assignment ensures:

- All events for a given partition (e.g., keyed by `locationId` for
  inventory) are consumed by the same pod, preserving per-partition ordering.
- Adding more pods scales by partition assignment, not by topic.
- Consumer group lag (`kafka.consumer.lag.records{group=admin-projection}`)
  is the primary catch-up indicator — see
  `architecture.md § Observability`.

## 3. Decision Table — Which Mechanism Applies

| Scenario | Mechanism | Behavior |
|---|---|---|
| REST mutation with same `Idempotency-Key`, same body | Redis | Replay cached 2xx |
| REST mutation with same `Idempotency-Key`, different body | Redis | 409 `DUPLICATE_REQUEST` |
| REST grant with new key after Redis TTL, same `(userId, roleId, warehouseId)` | Domain unique constraint | 200 with existing assignment row |
| REST POST `/users` with new key after Redis TTL, same `email` | Domain unique constraint | 409 `USER_EMAIL_DUPLICATE` |
| Kafka broker redelivers same `eventId` | PostgreSQL dedupe table | Skip, log, IGNORED_DUPLICATE |
| Kafka delivers fresh `eventId` with stale `occurredAt` (out of order) | last_event_at guard | Skip, log, IGNORED_DUPLICATE_LATE |
| Kafka DLQ replay (operator-initiated) within 30d | PostgreSQL dedupe table | Skip if already processed |
| Kafka DLQ replay after 30-day TTL purge | No dedupe row exists | Reprocess — accepted risk (see §3.1) |
| Two pods consume same partition briefly (rebalance) | Kafka partition assignment | One consumer at a time per partition |

### 3.1 30-day Replay Risk

After 30 days, the dedupe table has purged the row. An operator-initiated
replay older than 30 days would:

1. Re-insert the dedupe row with `outcome=APPLIED`
2. The `last_event_at` guard kicks in — most projection mutations are skipped
   because the target row's `last_event_at` is already newer
3. Append-only tables (`adjustment_audit`, `alert_log`) are protected by the
   PK constraint on `id = eventId` — duplicate row insert silently fails

Net effect: in v1, the 30-day replay risk is **mostly absorbed** by
last-write-wins + PK uniqueness. The exception is throughput counters, which
would double-increment if both the dedupe row and the projection state
expired (counters do retain `last_event_at`, but the increment-by-one path
does not consult it for duplicate-event protection — only for stale-order
protection). Mitigation: ops replaying messages older than 30 days must
verify counter ranges first; the read-model rebuild procedure
(`runbooks/read-model-rebuild.md`) is the safe alternative.

## 4. Idempotency-Key Lifecycle Guidance for Callers

For the admin UI:

1. Generate one `Idempotency-Key` (UUID) when the user opens a mutation form
2. Include it in the submission
3. If the user double-submits (clicks twice, network retry): same key →
   cached response replayed
4. If the user navigates away and reopens the form: a new key is generated
   → a new mutation is permitted (correct behavior)
5. On 503 PENDING: retry after 1s
6. On 409 `DUPLICATE_REQUEST`: surface to user as "this request appears to
   conflict with a previous submission — refresh and retry"

For ops scripts (e.g., bulk user import):

1. Generate one key per logical operation (per row in the import file)
2. Persist `(row_index, key)` alongside the import job state so retries
   reuse keys
3. On 409 `USER_EMAIL_DUPLICATE` or `ROLE_CODE_DUPLICATE`: log and skip;
   the row is already in the desired state from a previous run

## 5. Observability

| Metric | Description |
|---|---|
| `admin.idempotency.hit.rate` | % of mutating REST requests served from Redis cache |
| `admin.idempotency.mismatch.count` | `DUPLICATE_REQUEST` body mismatches (usually a caller bug) |
| `admin.idempotency.pending.count` | PENDING-state hits (rare; concurrent duplicate races) |
| `admin.event.dedupe.hit.count` | Events skipped by `eventId` conflict |
| `admin.event.dedupe.late.count` | Events skipped by last_event_at guard |
| `admin.event.dedupe.table.size` | Current row count (alert if unexpectedly large or shrinking outside the cron window) |
| `admin.projection.lag.seconds{source_service,topic}` | Event time → applied time |
| `admin.projection.dropped.count{reason=stale\|duplicate}` | Late-event drops + dedupe hits |
| `admin.projection.error.count{topic}` | Exception rate per consumer |

Log events (structured JSON, INFO level unless noted):

- `idempotency_cache_hit` with `{key, method, path}` — REST replay
- `idempotency_key_mismatch` (WARN) with `{key, method, path}` — body diff
  detected
- `event_duplicate_skipped` with `{eventId, eventType, consumer}` — Kafka
  redelivery
- `event_stale_dropped` with `{eventId, eventType, eventOccurredAt,
  rowLastEventAt}` — last-write-wins drop
- `projection_applied` (DEBUG) with `{eventId, eventType, target}` —
  successful projection (sampled at 1% in production)
