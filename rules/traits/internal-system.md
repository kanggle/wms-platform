# Trait: internal-system

> **Activated when**: `PROJECT.md` includes `internal-system` in `traits:`.

---

## Scope

**사내 임직원·운영자 전용** 시스템에 적용된다. 외부 고객/공개 인터넷 트래픽을 받지 않으며, 인증은 조직의 표준 SSO(OIDC IdP)를 통하고, 접근은 역할 기반(RBAC)으로 통제되며, 일상 작업의 상당 부분이 **관리자·운영자 워크플로**(결재·대결·위임, 운영 콘솔, 스케줄 작업)로 이루어진다.

핵심 긴장(stress axis)은 "외부 노출이 없다"는 신뢰 경계와 "내부자도 최소 권한으로만 행동한다"는 통제 경계의 결합이다. 외부 트래픽이 없다는 사실이 **내부 통제를 느슨하게 할 근거가 되지 않는다** — 내부자 위협·오작동·권한 오남용이 주된 위험 모델이다.

일반적으로 [`audit-heavy`](./audit-heavy.md)(운영 행위의 불변 기록)와 [`transactional`](./transactional.md)(결재/상태 전이 정합성)과 함께 선언된다. [`multi-tenant`](./multi-tenant.md)와의 공존은 허용된다(내부 사용이어도 조직 단위 격리가 필요한 경우 — taxonomy Incompatibilities). 도메인 고유의 RBAC·결재 규칙(예: 특정 도메인의 승인 단계 수, 권한 카테고리)은 이 trait이 아니라 해당 `rules/domains/<domain>.md`가 소유한다 — 이 파일은 **도메인 무관 공통 골격**만 정의한다.

---

## Mandatory Rules

### I1. SSO 단일 인증 (자체 인증 금지)
인증은 조직 표준 **OIDC IdP(SSO)** 를 통해서만 이루어진다.

- 각 서비스는 **OAuth2 Resource Server** 패턴으로 IdP의 JWKS 기반 서명(RS256 등)을 검증한다. 자체 로그인 폼·자체 비밀번호 저장·자체 토큰 발급을 두지 않는다 (IdP가 단일 발급 권위).
- 토큰의 테넌트/조직 claim(예: `tenant_id`)을 검증해 **자기 경계의 값만 통과**시킨다 (foreign-tenant 거부; `multi-tenant` 공존 시 그 trait의 격리 규칙과 공동 적용).
- 서비스 간(workload) 호출도 표준 메커니즘(client_credentials 등)으로 인증한다 — 비인증 내부 호출 경로를 두지 않는다.
- IdP 연동 세부는 공통 인증 canonical 소스를 따른다 (도메인별 IdP 클라이언트 설정은 해당 프로젝트 spec).

### I2. 외부 노출 금지 · 내부 네트워크 경계
시스템은 **공개 인터넷에 직접 노출되지 않는다**.

- 외부 도메인 라우팅·공개 API 게이트웨이 엔트리를 두지 않는다. 진입은 내부망/내부 게이트웨이/VPN/사내 네트워크 뒤로 한정한다.
- 백킹 인프라(DB, 캐시, 브로커 등)는 외부는 물론 불필요한 내부 노출도 최소화한다 (네트워크 경계는 설계 문서에 명시).
- "내부 전용"이 방화벽 한 겹에만 의존하지 않도록, 인증(I1) + 인가(I3)가 네트워크 경계와 **독립적으로도** 성립해야 한다 (defense-in-depth — 네트워크 침투 시에도 RBAC가 최후 방어선).

### I3. RBAC 권한 매트릭스 (fail-closed, 최소 권한)
접근은 **역할 기반 권한 매트릭스**로 통제된다.

- 역할 × 리소스 × 액션의 권한 매트릭스를 **사전 문서화**한다. 매트릭스에 명시되지 않은 접근은 **거부**한다 (fail-closed — 기본 허용 금지).
- **최소 권한**: 각 역할은 직무 수행에 필요한 최소 권한만 가진다. 광범위한 슈퍼유저 역할을 일상 운영에 사용하지 않는다.
- 역할·권한의 부여/회수는 **권한 변경 이벤트로 감사**된다 ([`audit-heavy`](./audit-heavy.md) A1과 교차).
- 인가 판정은 게이트웨이/필터/메서드 등 **모든 통과 레이어에서 일관**되게 적용한다 (한 레이어만 통과하면 우회되는 구조 금지).

### I4. Admin/Operator 워크플로 명시
운영 작업은 **명시적으로 정의된 워크플로**로만 수행된다.

- **운영 작업 카탈로그**: 어떤 관리자/운영자 작업이 존재하는지(잠금/해제/강제 로그아웃/데이터 정정/설정 변경/수동 개입 등) 목록화한다. 카탈로그에 없는 임의 운영 경로(직접 DB 수정 등)를 정상 경로로 두지 않는다.
- **승인 워크플로**: 위험 작업은 승인/대결/위임 상태기계를 거친다. 상태 전이·승인자 자격 규칙은 [`transactional`](./transactional.md)의 상태 전이/멱등 규칙과 공동 적용된다 (구체적 단계 수·라우팅은 도메인 spec 소유).
- **권한 분리(Separation of Duties)**: 한 주체가 요청·승인·실행을 모두 수행하지 못하도록 분리한다 (자기 승인 금지).

### I5. Scheduled Jobs 거버넌스
배치/스케줄 작업은 운영 관점의 거버넌스를 받는다.

- **멱등성·재실행 안전**: 동일 작업의 재실행이 중복 부작용을 내지 않는다 (재시도·중복 트리거 안전). 처리 의미론(대량 처리 자체)은 [`batch-heavy`](./batch-heavy.md) 공존 시 그 trait이 소유하고, 여기서는 **운영 거버넌스**(중복 실행 방지·모니터링·수동 트리거 통제)를 소유한다.
- **중복 실행 방지**: 다중 인스턴스 환경에서 분산 락/리더 선출 등으로 한 스케줄의 동시 중복 실행을 막는다.
- **모니터링·알림**: 작업 성공/실패/지연이 관측되고 실패가 알림으로 이어진다 (silent 실패 금지).
- **수동 트리거는 운영 작업**: 스케줄 작업을 수동으로 강제 실행하는 것은 I4 운영 작업 카탈로그에 속하며 감사된다.

### I6. Operational Traceability (운영 추적성)
"어떤 운영 행위가 언제 누구에 의해 수행됐는가"가 **추적 가능**해야 한다.

- 추적 대상 **운영 행위의 목록**(I4 카탈로그 + 권한 변경 + 수동 개입 + 스케줄 수동 트리거)을 정의한다 — 이 trait은 *무엇을 추적할지*를 정의한다.
- *어떻게 불변으로 기록·보존·조회 제한할지*(스키마/불변성/보존/메타 감사)는 [`audit-heavy`](./audit-heavy.md) A2~A10이 소유한다 — 중복 정의하지 않고 cross-ref한다.
- `audit-heavy`가 함께 선언되지 않은 internal-system 프로젝트라도, 운영 행위 추적은 **최소한의 구조화 기록**(행위자·시각·작업·대상·결과)을 남긴다.

## Forbidden Patterns

- ❌ **자체 인증 경로** (SSO 우회 로그인·자체 토큰 발급·비인증 내부 호출)
- ❌ **외부 공개 엔드포인트 / 공개 인터넷 직접 노출**
- ❌ **fail-open 인가** (권한 매트릭스 미정의 액션을 기본 허용)
- ❌ **일상 운영에 광범위 슈퍼유저 역할 사용** (최소 권한 위반)
- ❌ **인가가 단일 레이어에만 적용되어 다른 경로로 우회 가능**
- ❌ **카탈로그 밖의 임의 운영 경로** (운영자가 직접 DB 수정 등 비정의 개입)
- ❌ **자기 요청 자기 승인** (권한 분리 위반)
- ❌ **비멱등·중복 실행 방지 없는 스케줄 작업**
- ❌ **silent 실패하는 스케줄 작업** (모니터링·알림 부재)
- ❌ **운영 행위 무추적** (누가 무슨 운영 작업을 했는지 재구성 불가)

## Required Artifacts

1. **권한 매트릭스** — 역할 × 리소스 × 액션 표 (fail-closed 기본값 명시). 위치: `specs/` 하위 전용 문서 또는 도메인 spec.
2. **운영 작업 카탈로그** — 모든 admin/operator 작업 목록 + 위험 등급 + 필요 승인 단계.
3. **승인 워크플로 정의** — 상태기계 + 승인자 자격 + 권한 분리 규칙 (도메인 spec과 연계).
4. **SSO 통합 스펙** — IdP, 검증 방식(JWKS/RS256), 통과 tenant/조직 claim, workload 인증 방식.
5. **네트워크 경계 명세** — 내부 전용임을 증명하는 경계 다이어그램/정책 (외부 노출 경로 없음 확인).
6. **스케줄 작업 인벤토리** — 작업별 트리거·멱등성 전략·중복 실행 방지·모니터링/알림.
7. **운영 추적 대상 목록** — I6에서 추적할 운영 행위 (기록 방법은 `audit-heavy` 산출물로 위임).

## Interaction with Common Rules

- [`./audit-heavy.md`](./audit-heavy.md) — I6(무엇을 추적할지)과 audit-heavy A1~A10(어떻게 불변 기록할지)이 **분담**한다. internal-system은 운영 행위 대상을 정의하고, audit-heavy는 기록·불변성·보존·메타 감사를 정의한다. 중복 금지, cross-ref.
- [`./transactional.md`](./transactional.md) — I4 승인 워크플로의 상태 전이·멱등·정합성은 transactional 규칙과 공동 적용된다.
- [`./multi-tenant.md`](./multi-tenant.md) — 공존 허용(taxonomy Incompatibilities). I1의 tenant claim 통과 규칙은 multi-tenant의 격리 규칙과 공동 소유 — 내부 시스템이어도 조직 격리가 필요하면 둘 다 적용.
- [`./batch-heavy.md`](./batch-heavy.md) — 공존 시 batch-heavy는 대량 처리 의미론을, internal-system I5는 스케줄 작업의 **운영 거버넌스**(중복 방지·모니터링·수동 트리거 감사)를 소유한다 (이중 소유 회피).
- **공통 인증/플랫폼 규칙** — I1의 SSO·OAuth2 Resource Server·workload 인증은 `platform/` 인증 canonical 소스를 따른다 (이 파일은 trait 골격, 구체 IdP 설정은 프로젝트 spec).
- **도메인 규칙** — 도메인 고유의 RBAC 카테고리·결재 단계·운영 콘솔 요건은 해당 [`../domains/<domain>.md`](../domains/)가 소유한다. 이 trait은 도메인 무관 골격만 정의하므로, 도메인 spec이 이 골격을 특화·보강한다.

## Checklist (Review Gate)

- [ ] 인증이 표준 SSO(OIDC) 단일 경로이고 자체 인증/비인증 내부 호출이 없는가? (I1)
- [ ] tenant/조직 claim을 검증해 자기 경계 값만 통과시키는가? (I1)
- [ ] 공개 인터넷 직접 노출 경로가 없고 내부 네트워크 경계가 명시되어 있는가? (I2)
- [ ] 인증·인가가 네트워크 경계와 독립적으로도 성립하는가 (defense-in-depth)? (I2)
- [ ] 역할×리소스×액션 권한 매트릭스가 문서화되어 있고 기본값이 fail-closed인가? (I3)
- [ ] 일상 운영에 최소 권한 역할만 사용되며 권한 변경이 감사되는가? (I3)
- [ ] 인가가 모든 통과 레이어에서 일관되게 적용되는가 (우회 경로 없음)? (I3)
- [ ] 운영 작업 카탈로그가 존재하고, 카탈로그 밖 임의 운영 경로가 없는가? (I4)
- [ ] 위험 작업이 승인 워크플로(상태기계)를 거치고 권한 분리(자기 승인 금지)가 적용되는가? (I4)
- [ ] 스케줄 작업이 멱등·재실행 안전하며 중복 실행이 방지되는가? (I5)
- [ ] 스케줄 작업 성공/실패/지연이 모니터링·알림되는가 (silent 실패 없음)? (I5)
- [ ] 수동 트리거가 운영 작업으로 감사되는가? (I5)
- [ ] 추적할 운영 행위 목록이 정의되어 있는가 (기록 방법은 audit-heavy로 위임)? (I6)
- [ ] 금지 패턴(자체 인증, 외부 노출, fail-open 인가, 슈퍼유저 상용, 무추적 운영 등)이 코드베이스에 없는가?
